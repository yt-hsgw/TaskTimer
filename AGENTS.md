# AGENTS.md

## プロジェクト原則

このプロジェクトは、実務利用を想定したオフライン対応デスクトップアプリとして設計します。

- UIではなくデータモデルから設計する。
- Clean Architectureを使う。
- 意味が明確になる範囲でDDDの概念を使う。
- トランザクション境界を明示する。
- 状態変更と副作用を分離する。
- 推測した要件を実装しない。
- 重要な判断では、仮定、設計理由、トレードオフ、代替案を記録する。

## 必須ワークフロー

非自明な変更では、必ず次の順序で進めます。

1. 仕様を更新または確認する。
2. 設計を更新または確認する。
3. `docs/review/checklist.md` のレビューチェックリストを確認する。
4. 実装する。
5. テストまたは手動確認の結果を残す。

## セキュリティ基準

- デスクトップアプリの画面でもOWASP Top 10を意識する。
- すべての入力を検証する。
- 秘密情報をコードやリポジトリに保存しない。
- 将来のADRで明示承認されない限り、アプリ実行時の外部通信を追加しない。
- ユーザーのメモ本文をHTMLとして描画しない。
- タスク名、メモ本文、通知本文は、明示的に必要かつサニタイズ済みの場合を除きログへ出さない。

## アーキテクチャルール

- ドメインモデルは React、Tauri、SQLite、OS API に依存しない。
- Application Use Case がトランザクション境界を持つ。
- Infrastructure がRepository実装と副作用アダプターを持つ。
- OS通知はDBコミット後の副作用として扱う。
- アクティブタイマーの正はデータベースに置く。

## レビュー観点

レビューでは必ず以下を確認します。

- 破綻シナリオ。
- スケール問題。
- セキュリティ問題。
- 権限境界。
- テスト不足。
