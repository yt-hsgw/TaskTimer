# カレンダー期間・リサイズ設計レビュー

対象: GitHub #127 カレンダー項目の期間表示と端ドラッグ調整。

## 指摘事項

- 既存の開始予定/期限を予定期間へ流用すると、期限通知がリサイズ操作で移動するため意味が破綻する。
- UTC日時だけでは終日予定のローカル日付を安定して保持できず、タイムゾーン変更で表示日が変わる。
- UIだけで期間を伸ばすと再読み込み後に戻るため、Application層の検証とSQLiteトランザクションが必要である。
- HTML5 dragだけでは端のフォーカスとキーボード調整を提供しづらいため、リサイズハンドルを独立したボタンとして扱う必要がある。

## 判断

フォローアップ付き承認。

- 予定期間は期限から独立したローカル日付/時刻列で保存する。
- `ResizeScheduledWorkItem` をタスク/サブタスク共通の更新境界にする。
- カレンダー作成は `CreateScheduledTask` でタスクと期間を同一トランザクションに保存する。
- 期間リサイズで期限通知は変更しない。
- 重なり項目の横分割とDST実時間補正はMVP外とし、今後の利用状況で再評価する。

## 破綻シナリオ

- 開始端を終了以後へ移動する。
- 終了端を開始以前へ移動する。
- ポインター操作中に保存が失敗し、プレビューだけが残る。
- 月末や年末をまたぐ終日期間を縮める。
- 期間が表示範囲外から始まり、範囲内で終わる。
- 予定ブロック上の操作がセルのタスク作成操作へ伝播する。
- 削除済みまたはアーカイブ済み対象を更新する。

## スケール

- Read Modelは `scheduled_start_date <= range_end AND scheduled_end_date >= range_start` で範囲を絞る。
- タスク/サブタスクに期間範囲インデックスを追加する。
- 長大な期間による日別描画増加を抑えるため、期間を366日以内に制限する。
- 期間の重なり横分割は行わない。多数重なる場合の視認性は #72 の大量データ検証で確認する。

## セキュリティ

- ID、日付、時刻、終日フラグをApplication層で再検証する。
- CSSクラスへユーザー入力を渡さず、色は既存トークンだけを使う。
- タイトルとメモをHTMLとして描画せず、ログへ出さない。
- 外部通信、新しいOS権限、新しいTauri capabilityを追加しない。

## テスト計画

- Domain/Application: 逆転期間、15分刻み、終日、上限、対象ID検証。
- Infrastructure: タスク/サブタスク更新、範囲をまたぐRead Model、保存失敗時ロールバック。
- 繰り返し: 予定期間変更後も既存繰り返しルールが維持される。
- エクスポート: JSON/CSVへ全予定列が含まれる。
- Presentation: 週/日/月の期間表示、両端ハンドル、キーボード操作、狭幅でのはみ出しを確認する。

## 検証結果

- Rust全ターゲット: 91テスト成功。既存DB移行、タスク/サブタスク、繰り返し、エクスポートを含む。
- Clippy: 全ターゲット・全機能を警告エラー扱いで成功。
- Frontend: TypeScript型検査とVite本番ビルドに成功。
- Presentation: 週/日/月の期間表示、両端ハンドル、キーボード操作、表示領域からのはみ出しがないことを自動スクリーンショットで確認。
- Privacy: 実行時の外部通信API、ログ出力、リモートアセット、更新機能権限がないことを監査済み。

## 残余リスク

- OSやポインティングデバイスによるドラッグ感度の差は、Windows実機でリリース前に確認する。
- 同一時刻へ多数の予定が重なる場合は横分割しないため、後に描画された項目が前面へ出る。
- DST境界の経過実時間ではなく、保存したローカル壁時計時刻を表示する。

## 代替案

`scheduled_start_at` / `scheduled_end_at` をUTC RFC 3339で保存する案。

不採用理由: 外部同期を持たないローカルアプリでは、終日予定と壁時計時刻の維持を優先する。将来外部カレンダー同期を追加する場合は、タイムゾーンIDを含む別ADRで再設計する。
