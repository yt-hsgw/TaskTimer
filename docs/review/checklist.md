# 設計・実装レビューチェックリスト

## プロダクト

- 実装前に要件が書かれている。
- MVP範囲かMVP外かが明確。
- ユーザーに見える挙動が説明されている。
- 受け入れ条件がテスト可能。

## アーキテクチャ

- 挙動変更時にドメインモデルが更新されている。
- Clean Architectureの境界が保たれている。
- Use Caseがトランザクション境界を持っている。
- Infrastructureが副作用を持っている。
- 状態と副作用が分離されている。

## データ

- 新しいフィールドに検証ルールがある。
- マイグレーション方針が説明されている。
- インデックス影響が考慮されている。
- 削除挙動が明示されている。
- 単一アクティブタイマー制約が守られている。

## セキュリティ

- 入力検証が定義されている。
- ユーザー内容をHTMLとして描画していない。
- ユーザー内容をログに出していない。
- アプリ実行時の外部通信を追加していない。
- 新しい権限に理由がある。
- 秘密情報をコミットしていない。

## 公開運用

- README、LICENSE、SUPPORT、CONTRIBUTING、Release notesに矛盾がない。
- 外部利用者へ既知制限とサポート範囲が伝わる。
- Release workflowの権限が最小限である。
- Release artifactに秘密情報、DB、ログ、個人データが含まれない。
- 署名なし配布や自動更新なしの制限が明記されている。

## スケール

- 週カレンダークエリが日付範囲で絞られている。
- 大量のタスク/サブタスクが考慮されている。
- UI描画が全タイマー履歴の読み込みに依存していない。
- 通知再試行が無制限ループしない。

## 危険シナリオ

- タイマー開始中にアプリを終了する。
- タイマー開始中にOSがスリープする。
- サブタスクタイマー開始中に親タスクを削除する。
- DBコミット後に通知登録が失敗する。
- データベースマイグレーションが途中で失敗する。
- ユーザーが巨大なメモを作成する。

## レビュー出力

各レビューでは以下を記録する。

- 指摘事項。
- 破綻シナリオ。
- スケール懸念。
- セキュリティ懸念。
- テスト不足。
- 判断: 承認、フォローアップ付き承認、却下。
