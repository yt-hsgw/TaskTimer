# 操作・タイマー改善設計レビュー

対象: かんばんD&D、カレンダー移動、描画更新、独立ポモドーロ、タスクカウントダウン。

## 指摘事項

- かんばんの背面表示は `z-index` 不足だけでなく、状態列のスクロール領域によるクリッピングが原因である。
- カレンダーの期限と予定期間は別データであり、1つのD&D規則で同時更新すると通知意図が破綻する。
- 現行 `loadSnapshot` は更新対象外のRead Modelと通知同期まで再実行し、全画面再描画を誘発する。
- 現行ポモドーロ作業フェーズはタスクタイマー履歴と結合しているため、画面移動だけでは独立機能にならない。
- カウントダウンの終了時刻をReactだけで持つと、再起動、スリープ、設定変更、二重通知を正しく扱えない。

## 判断

フォローアップ付き承認。

- GitHub #138はPresentation内の変更で完結するため先行実装する。
- カレンダーは期限移動と予定期間移動を別Use Caseとして扱う。
- 全画面再描画は取得範囲を分割してから計測ベースでメモ化する。
- ポモドーロはタスク履歴を変更しない独立集約へ移行する。
- タスクカウントダウンは開始時の有効秒数と通知済み状態を永続化する。

## 破綻シナリオ

- 完了チェックのクリックがドラッグへ変わる。
- 予定期間移動が期限通知まで動かす。
- 非同期の古い再取得結果が新しい操作結果を上書きする。
- 独立ポモドーロと通常タイマーが同時に開始する。
- カウントダウン自動完了と手動停止が同時にコミットされる。
- スリープ復帰時に同じタイマー完了通知が複数回送られる。

## スケール

- `DragOverlay` は移動中カード1件だけを描画する。
- カレンダーは表示範囲内の項目だけを移動対象にする。
- 再描画計測は大量データfixtureで行い、全タスクの再取得回数とコミット時間を記録する。
- 1秒ごとの時計更新をApp全体のstateへ置かず、アクティブ行へ限定する。

## セキュリティ

- D&Dで受け取る対象IDと列ID、日付、時刻はApplication層で再検証する。
- タイトル、メモ、通知本文をHTML化または計測ログへ出さない。
- OS通知は既存のプライバシー設定と権限境界を使う。
- 外部通信、分析SDK、リモートアセット、新しいTauri権限を追加しない。

## テスト計画

- Presentation: カード全面ドラッグ、単純クリック、前面オーバーレイ、キーボードD&D。
- Calendar/Application: 期限移動と予定期間移動の分離、15分/1日刻み、失敗時復元。
- Performance: 完了、詳細保存、D&D、タイマーtickの再描画/クエリ回数。
- Domain/Infrastructure: 独立ポモドーロ移行、単一アクティブ制約、旧履歴保持。
- Timer/Notification: 開始時秒数スナップショット、再起動、スリープ、二重完了、通知OFF。

## テスト不足

- Tauri WebView上のPointer/Keyboard D&DはDOM単体テストだけでは操作感を保証できない。
- Windows実機のドラッグ感度とOS通知表示は最終確認が必要である。

## 先行実装の検証結果

- Frontend: TypeScript型検査とVite本番ビルドに成功。
- Presentation: 50タスク/200サブタスクのヘッドレスUIスモークで、通常クリック、カード本文からのドラッグ、`body`直下の前面オーバーレイ、移動先列強調、120ms保存遅延中の移動元非表示、保存後の列移動を確認。D&D操作は428msで1,500ms閾値内。
- Rust: 全ターゲット95テスト成功。
- Runtime privacy: 37ファイルを監査し、外部通信API、実行時ログ、リモートアセット、更新権限なし。
- Tauri: 既存の5173を避け、5174の一時設定で開発ビルドを起動確認。

## タスクカウントダウン実装前レビュー

- 判断: フォローアップ付き承認。既存の `timer_sessions` 集約を拡張し、別のアクティブタイマーテーブルは作らない。
- トランザクション境界: 開始時の有効秒数確定とセッション作成、自動完了判定とセッション停止をそれぞれ単一トランザクションにする。OS通知はコミット後に送信する。
- 互換性: `target_seconds` がnullの既存セッションとポモドーロ作業セッションは自動完了しない。
- スケール: 期限判定は単一アクティブセッションだけを読み、1秒更新はアクティブ行のローカルstateだけを更新する。
- セキュリティ: 既定秒数と対象固有秒数をApplication層とSQLite制約で60秒以上86,400秒以下に制限し、通知本文は既存表示タイプを再利用する。
- 残存リスク: DBコミットとOS通知の間でプロセスが終了すると通知を失う可能性がある。重複回避を優先し、永続再送キューは別Issueの対象とする。

## タスクカウントダウン実装後レビュー

- 判断: 承認。Issue #142の受け入れ条件を満たし、既存のClean Architecture境界と単一アクティブ制約を維持している。
- 境界: Domainは時間制約と完了理由、Applicationは入力検証と通知副作用、Infrastructureは開始時スナップショットと自動完了トランザクション、Presentationは表示と操作に限定されている。
- 互換性: 旧DBへ新しいセッション列とsingleton設定を追加する移行テスト、および旧セッションを自動完了対象外にするnull互換を確認した。
- 破綻シナリオ: 一時停止中の期限超過、再開後の休止時間控除、二重完了、通知OFF、通常タイマー/ポモドーロ相互排他を自動テストで確認した。
- 状態整合性: 起動・復帰・0秒到達時は期限判定を先にコミットし、その後でタスク一覧、カレンダー、実行状態を取得する。期限判定前の古いRead Modelが完了結果を上書きしない。
- スケール: 50タスク/200サブタスクのUIスモークで行内の開始、一時停止、再開、終了と配置を確認し、1秒更新で `App` / `TaskPanel` が再描画されないことを確認した。4操作を含む計測は1,367msで3,000ms閾値内だった。
- セキュリティ: 目標時間を1分以上24時間以内へ制限し、通知本文は既存プライバシー設定を再利用する。外部通信・新規権限・ユーザー本文のログ出力は追加していない。
- 検証: Rust全102テスト、Clippy、TypeScript/Vite本番ビルド、実行時プライバシー監査、UIスモークが成功した。
- テスト不足: Windows実機で0秒到達時の通知表示と、スリープ復帰直後の通知タイミングを最終確認する必要がある。
