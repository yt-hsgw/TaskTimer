# 共通タスク作成ダイアログ設計・実装レビュー

対象: GitHub #158

## 仕様・設計レビュー

- App Shellが作成プリセット、開閉、送信状態、作成エラーを所有する。
- 一覧、かんばん、`Ctrl+N` は通常作成プリセットを渡し、既存の `CreateTask` を使う。
- カレンダーのダブルクリック、Enter、ドラッグ範囲選択は予定プリセットを渡し、既存の `CreateScheduledTask` を使う。
- 通常作成と予定付き作成は同じダイアログを共有するが、Applicationのトランザクション境界は統合しない。
- サブタスク追加は親子関係を参照しながら操作するため、タスク詳細内のインライン入力を維持する。

## 指摘事項

ブロッキング指摘はない。実装レビュー中に、保存後の再取得で起点ボタンDOMが差し替わるとフォーカス復帰できない問題と、モーダル表示中に `Ctrl+B` で背面を変更できる問題を検出し、安定した起点属性による復帰と背面ショートカット抑止を追加した。

## 破綻シナリオ

- 保存に失敗する: ダイアログを閉じず、入力とカレンダー期間を維持してエラーをダイアログ内へ表示する。
- 保存ボタンを連打する: pending中は送信、閉じる、キャンセルを無効にする。
- 入力中にEscapeまたは背景を押す: 未保存入力があれば破棄確認を表示する。
- ダイアログ表示中に `Ctrl+N` を押す: 現在のプリセットと入力を初期化しない。
- ダイアログ表示中に `Ctrl+B` を押す: 左ペインを変更せず、背面状態を維持する。
- 作成中に選択リストが削除される: Application層の存在検証で失敗させ、入力を維持する。
- カレンダー期間の終了が開始より前になる: `CreateScheduledTask` の既存検証で拒否し、部分作成しない。

## スケール

ダイアログは表示中の1件分だけをstateとして保持し、タスク件数に比例するDOMや計算を追加しない。作成後の再取得範囲は既存のページング、カレンダー日付範囲、Repository Queryを維持する。401タスク、1,604サブタスク、12リストの標準UI計測で警告0件を確認した。

## セキュリティと権限境界

- タイトル、メモ、リストID、日付、時刻は既存Application Use Caseで再検証する。
- ユーザー入力はReactのテキスト描画と標準フォームvalueだけに渡し、HTMLとして描画しない。
- タイトル、メモ、作成エラーの入力内容をログへ出さない。
- 外部通信、OS権限、Tauri capability、ファイルアクセスを追加しない。
- 通常作成と予定付き作成は既存トランザクションを使い、Presentationで作成後更新を連結しない。

## トレードオフと代替案

各ビューでフォームを維持する案は、表示形式ごとに入力、エラー、フォーカス、保存挙動が分岐するため不採用とした。単一の新規バックエンドコマンドへ統合する案は呼び出し側が単純になるが、通常期限と予定期間の意味が混ざり、既存Use Caseの責務と検証が曖昧になるため採用しない。

## 判断

承認。タイムライン追加時は同じ `TaskCreatePreset` 境界を再利用し、タイムライン固有の初期期間だけをPresentationから渡す。
