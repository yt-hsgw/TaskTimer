# カレンダー予定ブロック移動設計レビュー

対象: GitHub #139 予定ブロック移動と期限調整操作の統合。

## 指摘事項

- UIから開始・終了を再送すると、古いRead Modelや改ざん入力で期間長を意図せず変更できる。
- 予定ブロック移動を既存の期限更新へ統合すると、期限通知まで暗黙に移動する。
- Repositoryの読み取りと更新を別トランザクションにすると、同時更新で古い期間長を使う競合が起きる。
- HTML D&Dのドロップ直後に仮位置を解除すると、DB再取得前の旧位置が一瞬再表示される。
- リサイズハンドルと本体ドラッグが同じ開始領域だと、端調整がブロック移動として保存される。

## 判断

承認。

- `MoveScheduledWorkItem` は移動先だけを受け取る専用Use Caseとする。
- Repositoryは現在期間の取得、Domainでの移動計算、予定期間列の更新を1トランザクションで行う。
- 期限、開始予定、通知ルールは移動commandの更新対象に含めない。
- UIは保存完了とRead Model再取得まで仮位置を保持し、失敗時だけDB位置へ戻す。
- 本体は移動、独立した両端ボタンはリサイズという操作境界を維持する。

## 破綻シナリオ

- 移動中に対象が削除またはアーカイブされる。
- 予定期間のない期限マーカーを予定移動commandへ渡す。
- 時刻あり予定へ時刻なし、終日予定へ時刻ありの移動先を渡す。
- 月末・年末またぎで終了日時が範囲外になる。
- 保存失敗後も仮位置やドラッグ中スタイルが残る。
- 月表示への移動で時刻あり予定が終日予定へ変わる。
- リサイズハンドルのpointer操作が本体dragへ伝播する。

## スケール

- 更新対象は主キー1件で、現在期間の取得と更新を同一トランザクション内で行う。
- 保存後はタスクページと表示範囲のカレンダーだけを再取得し、全Snapshotや通知同期を実行しない。
- UIの仮状態は移動対象1件だけを保持し、全項目の複製状態を作らない。
- 期間上限366日を維持し、1件の移動で無制限の日別セグメントを生成しない。

## セキュリティ

- 対象種別、ID、日付、時刻をApplication/Domainで再検証する。
- SQLのテーブル選択は列挙済みのタスク/サブタスク分岐だけにし、入力文字列をSQLへ連結しない。
- タイトル、メモ、親タスク名をログやdrag payloadへ含めない。
- 外部通信、新しいOS権限、新しいTauri capabilityを追加しない。

## テスト計画

- Domain: 時刻あり・終日、日/月/年またぎ、期間長維持、15分刻み、種別不一致。
- Application: IDと移動先の検証、Repository command境界。
- Infrastructure: タスク/サブタスク、予定なし、削除/アーカイブ、期限・通知非変更、トランザクション失敗。
- Presentation: 週/日/月D&D、15分丸め、キーボード、端リサイズ分離、仮位置と失敗復元。
- Performance: 更新commandと限定Read Modelだけが呼ばれ、通知同期を呼ばないことを大量データfixtureで確認する。

## 代替案

UIで現在期間との差分を計算し、既存の `ResizeScheduledWorkItem` に開始・終了を送る案。

不採用理由: 表示後にDBが更新された場合に古い終了日時で上書きし、移動とリサイズの監査可能な境界も失われる。

## 検証結果

- Rust全ターゲット: 99テスト成功。時刻あり/終日の日付またぎ、タスク/サブタスク、予定なし拒否、通知ルール非変更を含む。
- Clippy: workspace、全ターゲット、全機能を警告エラー扱いで成功。
- Frontend: TypeScript型検査とVite本番ビルドに成功。
- Presentation: 401タスク/1,604サブタスクの標準UI計測で、予定D&Dとキーボード移動、15分丸め、保存中の仮位置、限定Read Model再取得を確認。予定移動は374msで1,500ms閾値内。
- Notification boundary: 予定移動時に `update_task`、`update_subtask`、`sync_notifications`、OS通知登録処理を呼ばないことを確認。
- Privacy: 38ファイルを監査し、外部通信API、実行時ログ、リモートアセット、更新権限なし。

## 残余リスク

- OS/WebViewごとのHTML D&D感度は自動テストだけでは保証できないため、Windows実機で最終確認する。
- 同じ時刻に多数の予定が重なる場合は横分割しないため、移動先の選択性は既存制約を引き継ぐ。
