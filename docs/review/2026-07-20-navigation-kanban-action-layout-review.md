# 左ペイン・カンバン操作配置 設計レビュー

対象: GitHub Issue #178 / `docs/issues/070-navigation-kanban-action-layout.md`

## 指摘事項

### 1. カンバン列指定作成を2コマンドへ分けない

`CreateTask` の成功後に `MoveTaskToBoardColumn` が失敗すると、利用者が選んだ列と異なる場所へタスクが残る。Application Use CaseとRepositoryに列指定作成境界を追加し、SQLiteの同一トランザクションで作成する必要がある。

対応: `CreateTaskInBoardColumn` を追加し、列ID検証、列存在確認、タスクINSERTを1トランザクションへまとめる。

### 2. 追加スロットをSortableContextへ含めない

状態追加UIを状態列と同じ見た目にしても、永続化対象の列IDではない。Sortable itemやタスクdrop targetに含めると、並べ替えや移動先判定が不正になる。

対応: 追加スロットは `kanban-board` の末尾へ置くが、`SortableContext.items` は既存列IDだけに限定する。

### 3. 折りたたみ時の浮遊UIを閉じる

リスト編集フォームやポータル表示の縦三点メニューが開いたままセクションを閉じると、対象行が非表示なのに操作だけが残る。

対応: 折りたたみ時に作成、編集、メニュー状態を同時に閉じる。

### 4. 完了タスク全件削除を読み込み済み行へ限定しない

かんばんはカーソルページングされるため、Presentationの `completedRows` を順番に削除するとページ外の完了タスクが残り、途中失敗時には部分削除になる。

対応: `DeleteCompletedTasksInBoardColumn` がDB上の対象IDを取得し、既存のタスクグラフソフト削除を同一トランザクション内で全件へ適用する。削除件数を返し、0件でも成功として扱う。

### 5. ソートと永続順を分離する

列内D&Dによる順序保存は未実装であり、表示ソートから `sort_order` を更新すると一覧表示や別スコープの順序まで変わる。

対応: ソートは列コンポーネントの派生表示とし、既定順、期限、作成日、タイトルだけを選択可能にする。Repository更新は行わない。

## 破綻シナリオ

- 作成ダイアログ表示後に状態列が削除された場合、Repositoryの存在確認で保存を拒否し、入力とエラー表示を維持する。
- 状態名編集中に列D&Dを開始した場合、専用の最上部ハンドルだけがD&D listenerを持つため編集操作と競合しない。
- 列メニュー表示中に外側クリックまたはEscapeが発生した場合、メニューを閉じて縦三点ボタンへフォーカスを戻す。
- 全件削除の確認中または実行中はメニュー操作を閉じ、二重送信を防ぐ。
- 状態列が0件の場合も最右端の追加スロットを表示し、復旧できる。
- リストセクションを閉じた状態で左ペイン全体を縮小しても、既存のアイコン移動導線は維持する。

## スケール

- 追加操作は表示済みの状態列ごとに1ボタン増えるだけで、新しいQueryやタスク全件走査は増えない。
- 列指定作成は既存INSERT前の主キー存在確認だけを追加し、ページング済みRead Modelへ影響しない。
- 追加スロットをD&D itemへ含めないため、列数に対する衝突判定の性質は変わらない。
- 一括削除は対象完了タスク数に比例する。ローカルSQLiteの単一トランザクションで実行し、ページごとの追加Queryは行わない。
- ソートは表示中の最大200行に対する `O(n log n)` とし、全タスクの追加読み込みは行わない。

## セキュリティ

- 新しいコマンドは任意SQLやパスを受け取らず、既存タスク入力と状態列IDだけを受け取る。
- 状態列IDをApplicationで形式検証し、Infrastructureで有効列の存在を確認する。
- 外部通信、新しいOS権限、HTML描画、個人データのログ出力は追加しない。

## 権限境界

- Presentationは作成先の状態列IDを指定するだけとする。
- Applicationが入力検証とトランザクション境界を持つ。
- Infrastructureが状態列存在確認とINSERTを行う。
- Tauri commandはDTO変換とUse Case呼び出しに限定する。

## テスト計画

- Rust: 指定した状態列へタスクを作成できる。
- Rust: 存在しない、または削除済みの状態列では作成全体が失敗する。
- Rust: 完了タスク全件削除で対象列の完了タスクと配下データだけがソフト削除され、未完了タスクと他列が残る。
- TypeScript: 型検査とlintで列ID付き作成境界を確認する。
- UI手動確認: 左ペイン順序、折りたたみ、リスト追加位置、列追加位置、各列タスク追加、列メニュー、ソート、列幅ハンドル、列D&D、カードD&D。
- Playwright: デスクトップ幅と狭幅でスクリーンショットを確認し、文字切れ、重なり、不要な全体スクロールがないことを確認する。

## テスト不足

現状はReact component test基盤がないため、フォーカス復帰とPointer/Keyboard D&DのDOMイベント回帰は型検査、既存E2E相当の画面操作、手動確認を組み合わせる。将来はPresentation component testの導入を別Issueで検討する。

## 判断

フォローアップ付き承認。

条件:

- 列指定作成を同一トランザクションにする。
- 追加スロットをSortable/drop対象へ含めない。
- 実装後にRustテスト、フロントエンド静的検査、画面確認を行う。
