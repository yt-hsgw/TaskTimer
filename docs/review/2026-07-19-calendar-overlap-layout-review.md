# カレンダー重複予定レイアウト設計レビュー

対象: GitHub #147 重複する時間指定タスクを横並び表示する。

## 指摘事項

- 時間セルごとに全予定を再走査すると、予定数と表示時間数の積で計算量が増える。
- レーン番号をDBへ保存すると、表示期間や予測予定によって変わる派生値が永続状態へ混入する。
- 区間の終端を包含すると、終了と開始が同時刻の予定まで不要に横分割される。
- 単純にDOM順だけで横位置を決めると、Read Modelの返却順や予測予定の追加で左右へちらつく。
- 全予定を同じ `z-index` にすると、狭いレーンでフォーカスやリサイズハンドルが隣の予定に隠れる。

## 判断

承認。

- レーンはPresentationの導出状態とし、DB、Domain、Application、Repositoryを変更しない。
- 日付ごとに半開区間を安定ソートし、最小ヒープを使ったinterval partitioningで割り当てる。
- 連結した重複グループの最大レーン数をグループ全体へ適用し、途中でカード幅を変えない。
- 元予定と移動・変更予測を同じ入力として計算し、選択・フォーカス・予測を前面表示する。
- DOMへは数値のレーン情報だけを付与する。

## 破綻シナリオ

- `09:00-10:00` と `10:00-11:00` が2レーンになる。
- 完全重複した予定へ同じレーン番号が割り当てられる。
- 長い予定を介した連鎖重複の途中で `laneCount` が変わり、カード幅がずれる。
- 予測予定が入った瞬間に元予定のレーン順が非決定的に入れ替わる。
- 同じIDの表示用予測と保存済み項目がMap上で衝突する。
- 4件以上の重複でタイトルがリサイズハンドルを押し出す。
- 100件以上の同日予定で二乗計算になり、ドラッグ中に固まる。

## スケール

- 表示中の単日時刻あり予定だけを日付単位に分類する。
- ソートと2つの最小ヒープで `O(n log n)`、レイアウトMapで `O(n)` とする。
- `useMemo` の依存を表示予定配列へ限定し、時刻行ごとにレーン計算を繰り返さない。
- レーン計算中にDB、Tauri command、React state更新を実行しない。

## セキュリティ

- タイトルやメモをdata属性・ログへ複製しない。
- ユーザー文字列をHTMLとして挿入しない。
- 外部通信、新しいTauri権限、DB変更を追加しない。
- 移動・変更後の日時は既存Use Caseで検証する。

## テスト不足と確認項目

- 完全重複、部分重複、境界一致、連鎖重複のレーン番号と幅。
- タスクとサブタスクの混在。
- 各カード中心のヒットテストとキーボードフォーカス。
- 非重複予定の全幅維持。
- D&D、キーボード移動、期間端リサイズ、ドラッグ範囲作成の回帰。
- 標準大量データプロファイルでの描画時間とWARN件数。

## 代替案

既存のカレンダー全体を外部ライブラリへ置き換える案。

不採用理由: オフライン実行自体は可能だが、期限マーカー、サブタスク、予測表示、独自Read Modelとの再統合範囲が大きく、今回のPresentation内レイアウト変更に対して過剰である。

## 残余リスク

- 同時刻に多数の予定がある場合は文字領域が小さくなる。カード選択とアクセシブル名を維持し、一覧ポップアップやレーン拡張は実利用結果を見て別Issueで判断する。
- Windows WebView2でドラッグ画像と狭いレーンのヒット領域に差がないか実機確認が必要になる。
