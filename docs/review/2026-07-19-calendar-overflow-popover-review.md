# カレンダー省略予定一覧ポップアップ設計レビュー

対象: GitHub #148 「他 N 件」から当日の予定一覧を表示する。

## 指摘事項

- ポップアップ用に別クエリを追加すると、表示中データとの順序・更新タイミングが分かれ、不要なDBアクセスが発生する。
- 予定を対象IDだけで一意化すると、同じタスクの予定・期限・実行中マーカーが欠落する。
- viewport基準で固定配置すると、中央ペイン外や右詳細ペインへ重なりやすい。
- ポップアップ内のイベントがカレンダーへ伝播すると、範囲作成やD&Dを誤って開始する。
- 開閉時のフォーカス管理がないと、キーボード利用者が現在位置を失う。

## 判断

承認。

- 既存の日付レイアウトへ、その日の全予定を表示順どおり保持する。
- ポップアップ内容は現在の `WeekCalendarItem[]` から導出し、DB・Use Caseを追加しない。
- 中央ペイン基準の絶対配置を上下左右へ制約し、一覧だけをスクロールする。
- ポップアップ内でポインターイベントを停止し、予定行は詳細選択専用にする。
- 日付・件数をアクセシブル名に含め、開閉時のフォーカスを明示管理する。

## 破綻シナリオ

- 右下セルから開いた一覧が中央ペインの外へ切れる。
- 予定更新後も削除済み行が残り、選択時に詳細取得へ失敗する。
- 同じ対象のマーカーがReact key衝突で消える。
- `Escape` で閉じても起点へ戻れず、次のTab位置が不定になる。
- ポップアップの行をドラッグして予定期間が移動する。
- カレンダー本体のスクロール後もポップアップが元の座標へ残る。
- 1日に数百件あると中央ペイン全体の高さが増える。

## スケール

- 日付レイアウトは既存の表示範囲内データだけを対象とする。
- カレンダー本体は3レーンのままにし、ポップアップだけ最大420pxの内部スクロールにする。
- 開いている1日分だけを描画し、ポップアップ表示のためのTauri commandは発行しない。
- 予定数が実測上問題になる場合は、一覧仮想化を別Issueで検討する。

## セキュリティ

- ユーザー文字列はReactのテキストノードとして描画し、`dangerouslySetInnerHTML` を使わない。
- タイトル、親タスク名、メモをログやDOMのdata属性へ複製しない。
- 外部通信、Tauri capability、OS権限、DBスキーマを追加しない。
- 一覧からは既存詳細表示のみを呼び、更新系Use Caseを直接実行しない。

## テスト不足と確認項目

- 月・週・日表示での全件数と表示順。
- サブタスク、複数マーカー、複数日予定の行表示。
- 行選択、閉じる、外側クリック、`Escape`、フォーカス復帰。
- 右端・下端・狭いペインでの配置制約。
- ポップアップ操作時のcommand 0件と範囲作成・D&Dの回帰。
- スモークと標準大量データでの描画時間と警告件数。

## 代替案

React Portalでアプリ最上位へモーダルダイアログを表示する案。

不採用理由: 見切れ対策は容易だが、日付セルとの位置関係が失われ、右詳細ペインを含むアプリ全体を覆う。今回は中央ペイン内で予定を確認しながら操作を継続できる非モーダル表示を優先する。

## 残余リスク

- 予定数が極端に多い場合、ポップアップを開く瞬間のDOM生成時間が増える。標準大量データで計測し、必要なら仮想化を追加する。
- Windows WebView2でフォーカス復帰とスクロールイベントの順序に差がないか、公開前の実機確認対象とする。
