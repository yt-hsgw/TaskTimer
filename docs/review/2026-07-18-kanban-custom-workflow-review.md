# かんばんカスタム状態レビュー 2026-07-18

対象: GitHub #126 カスタム状態、列/カードのドラッグ&ドロップ、列ごとの完了セクション。

## 判断

フォローアップ付き承認。

業務状態と完了ライフサイクルを分離し、Domain/Application/Infrastructure/Presentationの依存方向を維持している。列削除と所属タスク移動、列並べ替え、カード移動はSQLite Repositoryのトランザクション内で完結する。

## 指摘事項

- `tasks.status` は既存画面との互換用に残るため、`board_column_id` / `lifecycle_status` との同期箇所を今後も増やさない設計が必要。
- かんばんカードは既存Read Model上限の200件までであり、大量カードの仮想スクロールは未実装。
- 列内カード順のドラッグ並べ替えは本Issueの対象外で、カード移動は列IDだけを保存する。

## 破綻シナリオ

- 列削除時に所属する完了タスクの `completed_at` が失われる。
- 初期列を削除した後、新規タスクが削除済み列へ紐づいて画面から消える。
- タイマー開始時に互換 `status` だけが進行中となり、かんばん列と不一致になる。
- 並べ替え要求に列IDの欠落、重複、削除済みIDが含まれ、順序が部分更新される。
- 旧DBの移行が途中で止まり、`board_column_id` がNULLのまま残る。

上記は、移動先必須の列削除トランザクション、有効な初期列の選択、タイマー開始時の列同期、全列ID一致検証、冪等なNULLバックフィルで対処した。

## スケール懸念

- `board_columns` の件数集計は `tasks_board_column_lifecycle_idx` を使う。200タスク、600サブタスク、1,001タイマー履歴の検証DBでは1ms未満だった。
- 状態列数の上限は設けていない。列が増える場合は中央ペイン内の横スクロールで表示する。
- 200件を超えるカード表示は既存Read Model上限に依存するため、仮想スクロールとページングを別Issueで扱う。

## セキュリティ懸念

- 状態名はtrim、80文字上限、制御文字禁止をDomainで検証し、SQLiteでも空文字と長さを制約する。
- 状態名、タスク名、メモ、タグはReactのテキストとして描画し、HTMLとして解釈しない。
- DnDライブラリはローカルバンドルされ、新しいTauri権限、外部通信、秘密情報、ユーザー内容のログ出力を追加していない。
- 外部通信監査は37ファイルを確認して成功した。

- スクリーンリーダーごとの読み上げ品質は未確認。
