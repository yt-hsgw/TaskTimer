# Windows復帰回帰検証レビュー

## 対象

- GitHub Issue #58
- タイマー開始・一時停止中のDB再接続とwall-clock経過
- 復帰相当の通知再同期と重複防止
- GitHub-hosted Windows VM回帰workflow

## 指摘事項

- Linux上の通常CIだけではWindows向け依存を含むSQLite・Use Case回帰を保証できない。
- 開始中タイマーのDB再接続はテスト済みだが、一時停止中に長時間経過して再接続する破綻シナリオが固定されていない。
- GitHub-hosted runnerを実際にスリープさせるとrunner接続自体を失うため、安定した自動検証には使えない。

## 設計判断

- ドメインモデル、Repository境界、トランザクション境界は変更しない。
- 一時停止中のDB再接続と時刻ジャンプを既存の`FixedClock`で再現し、停止区間が経過時間へ混ざらないことをRepositoryテストへ追加する。
- Windows runnerでRustテスト全体を実行し、Windows向け依存を含むコンパイルと回帰を継続確認する。
- 実電源イベント、WebView2フォーカス復帰、OS通知表示はRelease前手動確認として維持する。

## 代替案

Windows runnerをPowerShellからスリープさせる。

不採用理由:

- runnerとの通信断によりworkflow自体が失敗またはタイムアウトし、再現可能な検証にならない。
- GitHub-hosted環境の電源管理権限やハイパーバイザー挙動は利用者のWindows端末と一致しない。

## 破綻シナリオ

- 一時停止中のスリープ時間が作業時間へ加算される。
- 再接続後にアクティブタイマーが消える、または複数になる。
- 復帰相当の再同期で同じ通知を複数送信する。
- Windows CI成功を実際のOSスリープ確認済みと誤認する。

## スケール懸念

- Windows runnerでRustテスト全体を追加実行するためCI時間は増えるが、アプリ実行時性能やSQLiteクエリ件数には影響しない。
- Rustキャッシュを共有し、Pull Requestと手動実行に限定する。

## セキュリティ懸念

- Tauri権限、Windows電源管理権限、外部通信は追加しない。
- テストデータは一時ディレクトリ内の固定値だけを使い、利用者DB、タスク名、通知本文をartifactやログへ保存しない。
- workflow権限は`contents: read`だけとする。

## テスト結果

- 開始中タイマーがDB再接続後も同じIDと対象で復元され、長時間差分を含めて停止できる既存テストを確認した。
- 一時停止中タイマーがDB再接続後も一時停止状態で復元され、9時間58分の停止区間を除外して120秒で確定するテストを追加した。
- 復帰相当の通知同期が期限到来通知を1回だけ送信し、次の通知を再予約する既存テストを確認した。
- ローカルのRustテスト95件、Clippy、TypeScript/Vite build、npm監査、実行時プライバシー監査が成功した。
- GitHub-hosted Windows VMの結果はPull Requestの `Windows復帰回帰検証` とIssueコメントへ記録する。

## 判断

承認。実スリープを自動保証範囲から除外し、Release前手動確認を残すことを条件とする。
