# 表示範囲単位のRead Model更新 実装レビュー

対象: GitHub Issue #140、`docs/issues/056-scoped-ui-refresh.md`。

## 指摘事項と対応

- 操作後にタスク、リスト、かんばん、タグ、カレンダー、タイマー、設定、通知を一括取得していた。用途別更新関数とコマンド別無効化表へ分割した。
- 非表示中のタスク変更でカレンダーと、かんばん列件数が古くなる。dirty世代を記録し、次回表示時にだけ再取得する。
- 画面遷移と保存完了が競合すると、古いビュー条件の取得結果が現在画面を上書きし得た。保存完了時点の最新更新関数を参照し、各Read Modelのリクエスト世代番号でも古い結果を破棄する。
- 単一の `isMutating` が全画面を無効化していた。ナビ、詳細、かんばん、カレンダー、設定へ操作中状態を分けた。
- 親Appのstate更新で無関係なペインも再描画されていた。取得範囲を分けた後、主要ペインを `React.memo` でメモ化し、commit単位の描画プローブで検証した。

## 判断

承認。

Application Commandのトランザクション境界、SQLiteを正とする方針、通知のDBコミット後副作用は変更していない。PresentationのQuery実行範囲と表示stateだけを変更している。

## 破綻シナリオ

- 保存中に別ビューへ移動し、古い取得結果が現在ビューを上書きする。
- 複数タスクの操作が並行し、遅いQueryが新しい結果を上書きする。
- 非表示カレンダーを更新せず、次回表示でも古い期限や状態を出す。
- 通知設定変更後にOS登録を更新せず、表示内容または通知ON/OFFがずれる。
- 一部Queryだけ失敗した後、成功済みCommandをUIが未反映のまま扱う。

対応として、最新ビュー参照、用途別リクエスト世代、dirty世代、エラー表示と明示的な全再読み込みを実装した。

## スケール

- 401タスク、1,604サブタスク、12リストで、操作後のタスクページだけを再取得した。
- お気に入り49ms、完了49ms、詳細保存33ms、かんばんD&D 524msで、すべて既定閾値内だった。
- 同一ナビ再選択はQueryなし、主要コンポーネントの再描画なしで30msだった。
- かんばんD&D中と保存後に、左ナビ、カレンダー、設定の無関係な描画がないことを確認した。

## セキュリティ

- 外部通信、分析SDK、リモートアセット、新しいTauri権限は追加していない。
- 描画プローブはコンポーネント名と回数だけを保持し、タスク名、メモ、通知本文を記録しない。
- 入力値とIDの検証、Repository境界、OS通知権限境界は既存Application/Infrastructureへ委譲したままである。

## 検証結果

- Frontend: TypeScript型検査とVite本番ビルドに成功。
- Presentation smoke: 50タスク、200サブタスクの13計測が成功、警告0件。
- Presentation standard: 401タスク、1,604サブタスクの14計測が成功、警告0件。
- Rust: 全ターゲット95テスト成功。
- 差分形式: `git diff --check` 成功。

## テスト不足と残余リスク

- Tauri WebView実機での体感上のちらつきは、ヘッドレスChromeのDOM/描画回数だけでは完全には保証できない。
- Application Command成功後にQueryだけ失敗するケースは既存表示保持と再読み込みで復旧するが、操作別の自動再試行は行わない。
- 今後Read Modelを追加する際、コマンド別無効化表への登録漏れをレビューで確認する必要がある。

## 代替案

全データを再取得したまま、すべてのコンポーネントを `React.memo` で囲む案。

不採用理由: DB Queryと通知同期の重複が残り、毎回新しい配列参照を渡すためメモ化も安定して効かない。
