# タスク表示色分離 設計レビュー

対象: GitHub Issue #159 / `docs/issues/065-task-display-color.md`

## 指摘事項

- `tasks.color_token` はnullableとし、nullを「所属リスト色を継承する」というドメイン上の値として扱う。
- 色変更は既存の `UpdateTask` トランザクションへ含め、所属リスト変更と同時に行われても中間状態を作らない。
- Calendar Read Modelは本文用の実効タスク色と左端用の所属リスト色を別フィールドで返す。
- サブタスクに色列を追加せず、親タスクの実効色をクエリで解決する。
- JSON/CSVにはnullableな明示色を出力し、継承と明示指定を区別する。

## 設計理由

nullable継承により、個別色を設定していないタスクはリスト色の変更へ追従できる。色の検証はDomain/Application/DBの各境界で行い、Presentationへ任意のCSS値を流さない。Read Modelで実効色を解決するため、Presentationがリストと親タスクを追加走査せずに描画できる。

## 代替案とトレードオフ

代替案は、タスク作成時にリスト色を必須値としてコピーする方式である。読み取りは単純になるが、リスト色変更へ追従せず、継承か明示設定かも区別できないため採用しない。nullable方式はSQLの `COALESCE` とAPI項目が増えるが、利用者の意図を保持できる。

## 破綻シナリオ

- 旧DBに色列がない状態で起動すると読み取りが失敗するため、冪等な列追加マイグレーションを先に実行する。
- 期限変更や所属リスト変更の更新入力で現在色を渡さないと明示色が消えるため、更新Draftでは色を必須項目として扱う。
- 削除済みまたは欠損したリストとの結合では実効色を解決できないため、既存のリスト外部キーとソフト削除制約を維持する。
- 許可外トークンがDBへ入るとCSSクラスが不定になるため、Domain検証とCHECK制約の両方で拒否する。

## スケール懸念

色解決は既存Calendarクエリ内の `COALESCE` だけで完結し、追加クエリやN+1を発生させない。nullable列は絞り込みや並べ替えに使わないため、新規インデックスは不要と判断する。

## セキュリティと権限境界

- 色は6種類の固定トークンだけを受け入れ、任意CSS、URL、HTMLを保存しない。
- 新しいOS権限、外部通信、ログ出力は追加しない。
- タスク名やメモなどユーザー内容の扱いは変更しない。

## テスト不足への対応

- 旧DBマイグレーションとnull初期値。
- 許可値および不正値の拒否。
- 明示色の保存、継承への解除、所属リスト変更時の保持。
- 親タスクとサブタスクのCalendar Read Model色分離。
- JSON/CSVでのnullable色出力。
- フロントエンド型検査とUI操作確認。
