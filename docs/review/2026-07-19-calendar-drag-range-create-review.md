# カレンダードラッグ範囲作成設計レビュー

対象: GitHub #146 ドラッグ範囲から予定付きタスクを作成する。

## 指摘事項

- pointerupだけで即時保存すると、タイトルや所属リストを確認できず、誤ドラッグで不要なタスクが作成される。
- 空きセル、予定本体、期限マーカー、リサイズハンドルのpointerdownを共通処理にすると、移動・調整・新規作成が競合する。
- React stateだけをpointerupの正にすると、同一フレームのpointermove結果を取りこぼす可能性がある。
- 時間セル単位で期間を決めると、15分単位の選択と複数行にまたがる選択を表現できない。
- 月/上部予定行で日付セルごとに独立した仮要素を置くと、連続範囲に見えない。
- クリックとドラッグのしきい値がないと、ダブルクリック作成やスクロール意図を誤って範囲作成として扱う。

## 判断

承認。

- 範囲選択はPresentationの一時状態とrefへ保持し、pointerup時に既存作成フォームへ変換する。
- 保存は既存 `CreateScheduledTask` のみを使い、DB、Repository、通知境界を変更しない。
- 空きセル背景からだけ選択を開始し、既存予定と操作ボタンの子要素では開始しない。
- 日/週の時間軸は開始した日列内で15分単位、月/上部予定行は包含日付範囲として正規化する。
- 6px以上の移動後だけ範囲作成を確定し、pointer cancel、表示切替、フォーム表示で一時状態を解除する。
- 仮表示は `pointer-events: none` とし、既存D&Dやリサイズのヒットテストを遮らない。

## 破綻シナリオ

- 予定ブロックの余白を押したつもりが背面セルで新規選択を開始する。
- pointermove直後のstate更新前にpointerupし、1つ前の範囲でフォームを開く。
- Pointer Capture中に画面外へ出てpointerupし、選択状態が残る。
- 上方向または左方向の選択で開始・終了が逆転する。
- 22時台の下端を越えて23時より後の不正時刻になる。
- 月表示で前月・翌月の日付セルをまたいだ範囲が誤った年になる。
- 選択プレビューが既存の予定クリックやリサイズハンドルを覆う。
- 選択中にビュー切替やナビゲーションを行い、別画面へ仮表示が残る。

## スケール

- 時間選択は対象日1件、日付選択は表示中の最大42セルだけを判定する。
- pointermoveごとにDBアクセスやRead Model再取得を行わない。
- 月範囲プレビューは表示日ごとの真偽判定に留め、タスク一覧やタイマー履歴を複製しない。

## セキュリティ

- 座標から生成した日付・時刻も既存Application/Domainで再検証する。
- タイトル、メモ、親タスク名をログへ出さない。
- 文字列をHTMLとして挿入しない。
- 外部通信、新しいTauri capability、OS権限を追加しない。

## テスト計画

- ヘッドレスChromeで空き時間軸を実ポインター操作し、15分単位のプレビューとフォーム初期値を確認する。
- 上方向ドラッグと営業時間下端で正規化・クランプを確認する。
- 月表示と上部予定行で複数日終日範囲を確認する。
- pointerup前にTauri commandが呼ばれず、フォームキャンセル後も呼ばれないことを確認する。
- 予定本体D&D、端リサイズ、ダブルクリック、`Enter` の既存検証を継続する。
- TypeScript型検査、Vite本番ビルド、Rust全テスト、Clippy、runtime privacy監査を実行する。

## 代替案

カレンダーライブラリへ置き換え、標準のselect interactionを使う案。

不採用理由: 現在の期限マーカー、予定期間、タイマー、サブタスク、限定Read Model更新を再統合する影響が大きい。今回の選択状態は既存Presentation境界へ限定して追加できる。

## 残余リスク

- WebView2とChromiumでPointer Captureの感度が異なる可能性があるため、Windows実機確認は最終的に必要になる。
- タッチ操作と自動スクロールは初回範囲外であり、デスクトップのマウス操作を優先する。
