# ワークスペース表示切り替えとローカル検索レビュー

対象: GitHub #157

## 仕様・設計レビュー

- 左ペインの対象範囲と中央ペインの表示形式を別stateとして扱う。
- 実装済みのリスト、カンバン、カレンダーだけを表示する。タイムラインはGitHub #160へ分離する。
- 検索はApplicationの読み取りUse CaseとSQLite Repositoryで実行し、Reactの読込済みページだけを検索対象にしない。
- 検索結果からは親タスクIDを使って詳細を再取得し、一覧ページ外のタスクとサブタスクも開けるようにする。
- カレンダーは一覧と同じスコープをRepositoryへ渡し、Presentationで全件を後処理しない。

## 指摘事項

ブロッキング指摘はない。既存の `ui_preferences.last_view` は後方互換のため残しており、今日・お気に入りと表示形式の組み合わせまでは再起動後に復元しない。この永続化を追加する場合は設定スキーマのマイグレーションとして別Issueで扱う。

## 破綻シナリオ

- 連続入力中に古い検索応答が戻る: request IDが一致しない応答を破棄する。
- `%` や `_` を入力する: SQLワイルドカードではなく入力文字としてエスケープする。
- 削除済み、アーカイブ済み、削除済みリストのデータが残る: Repositoryの検索条件で除外する。
- 一覧ページ外のサブタスクを選ぶ: 親タスクIDからタスクツリーを読み直して詳細を開く。
- 選択中の表示形式を再度押す: state更新と再取得を行わない。
- 対象範囲を切り替えた直後にカレンダーを開く: 同じスコープをカレンダーQueryへ渡す。

## スケール

部分一致検索はSQLiteの通常 `LIKE` であり、件数増加時は全件走査がボトルネックになり得る。現段階では入力120文字、結果50件、200msデバウンスで上限を設ける。標準大量データで閾値を超える場合はSQLite FTS5と同期用仮想テーブルを代替案として検討する。

## セキュリティと権限境界

- 検索語はバインドし、`LIKE` のワイルドカードをエスケープする。
- タイトル、メモ、タグ、検索語をログへ出さず、外部通信を追加しない。
- Reactのテキスト描画を使い、ユーザー入力をHTMLとして挿入しない。
- 新しいOS権限、Tauri capability、ファイルアクセスは追加しない。
- 読み取りUse Caseは更新トランザクションと通知副作用を持たない。

## テスト

- Rust: タスク名、メモ、タグ、サブタスク検索、ワイルドカード、アーカイブ除外、入力上限、スコープ付きカレンダー。
- TypeScript: production build。
- UI: READMEスクリーンショット生成で表示切り替え、検索結果、詳細表示を確認。
- 性能: 大量データUI smoke profileで警告0件。
- 静的確認: `cargo fmt`、`cargo clippy`、`git diff --check`、runtime privacy auditを実行する。

## トレードオフと代替案

通常 `LIKE` は追加テーブルを必要としない一方、FTS5より大規模検索に弱い。FTS5は高速だが、タスク、サブタスク、タグ変更時の同期とマイグレーションが増えるため現段階では採用しない。React側の配列検索は単純だがページング外を見落とすため不採用とする。
