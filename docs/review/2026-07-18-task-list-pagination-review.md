# タスク一覧ページング設計・実装レビュー

対象: GitHub #131 タスク一覧の200件上限をページングで解消する。

## 指摘事項

- 完了状態と並び順は更新されるため、状態変更後に古いカーソルを再利用すると重複・欠落が起こる。
- タスクツリーと一覧行を別ページとして取得すると、一覧から詳細を開けない不整合が起こる。
- 今日、お気に入り、タグを読み込み済み200件だけで絞ると、DB全体の対象タスクを欠落させる。
- 追加読み込み応答がビュー切替後に到着すると、異なるスコープのデータが混ざる。
- 5,000件を一度にDOMへ追加すると、到達性は直っても操作性能が退化する。

## 判断

承認。

- `ListTaskPage` を読み取り専用Application境界とし、スコープ、日付、上限、カーソルを検証する。
- Repositoryは接続ロック内でページIDを確定し、同じID集合からタスクツリーと一覧行を返す。
- カーソルは完了区分、並び順、作成日時、IDで安定化する。
- 更新後は現在の読み込み件数まで先頭から再取得し、更新前カーソルを破棄する。
- Presentationは200件ずつ追記し、既存行、選択、展開、スクロール位置を維持する。
- カーソル順専用インデックスを追加し、全ページ巡回の全件ソートを避ける。
- 現行UIが使わなくなった旧全リスト一括Read Modelを性能ゲートから外し、5,000件カーソル巡回へ置き換える。

## 破綻シナリオ

- 同じ`sort_order`と`created_at`を持つタスクがページ境界に並ぶ。
- 追加読み込み中に別ビューへ切り替える。
- 読み込み済みタスクを完了、再開、削除、別のかんばん列へ移動する。
- 今日の対象が親タスク期限ではなくサブタスク期限だけで決まる。
- 未完了の最終ページと完了の先頭ページが同じカーソル境界に入る。

## スケール

- 初期描画と1回の追加は最大200件に制限する。
- 5,000件DBは性能計測で200件カーソルを最後まで巡回し、重複・欠落を検査する。
- DB総件数と読み込み済み件数を分離し、未取得データをゼロ件と誤表示しない。
- ページ順インデックスには追加の保存領域と書き込み更新コストがあるが、タスク一覧の読み取り頻度と到達性を優先する。

## セキュリティ

- SQL断片を受け取らず、列挙済みスコープとバインド変数だけを使う。
- ID、日付、ページ上限、カーソル形式をApplication境界で検証する。
- タスク名、メモ、タグ名、カーソルをログへ出さない。
- 外部通信、Tauri capability、OS権限を追加しない。

## テスト

- Repository: 同順位、完了境界、複数ページの重複・欠落を検査する。
- Repository/Application: 今日、お気に入り、タグのDBスコープと不正入力拒否を検査する。
- Presentation: 401件から初期200件、追加200件を描画し、選択、展開、詳細表示の維持を検査する。
- 性能: 5,000件の全ページ巡回とheadless Chromeの追加読み込みを閾値付きで計測する。

## 残余リスク

- SQLiteファイルを別プロセスが直接更新する運用はサポートしないため、外部書き込みをまたぐスナップショット整合性は保証しない。
- 5,000件から特定タスクへ一度で移動する検索UIは本Issueの範囲外である。
- Tauri IPCとWindows WebView2を含む実機の追加読み込み時間は、GitHub-hosted headless Chrome計測の対象外である。

## 検証結果

- Rust: 全94テスト成功。Clippyは全ターゲット・全機能を警告エラー扱いで成功。
- SQLite standard: 5,000タスクを25ページで巡回し、重複・欠落なし、107ms、WARN 0。
- Presentation standard: 401タスクの初期200件と追加200件を描画し、追加971ms、選択・展開・詳細・スクロール位置維持、WARN 0。
- Frontend: TypeScript型検査、Vite本番ビルド、READMEスクリーンショット生成に成功。
- Privacy: 実行時外部通信、ログ、リモートアセット、更新機能権限の静的監査に成功。

## 代替案

仮想リストだけを導入する案。

不採用理由: DOM数は削減できるが、Repositoryが返さない201件目以降へ到達できない問題を解消しない。
