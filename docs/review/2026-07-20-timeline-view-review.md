# タイムライン表示 設計レビュー

対象: GitHub #160

## 結論

初回は既存 `ListTaskPage` を入力とするPresentationの読み取り専用射影として実装する。新しいRepository Query、DBスキーマ、更新Use Caseは追加しない。日、週、月ごとに固定長の表示窓を設け、既存の最大200件ページングと追加読み込みを維持する。

## 仕様とデータ境界

```mermaid
flowchart LR
  Nav["左ペインの対象範囲"] --> Query["ListTaskPage Query"]
  Query --> Page["TaskPage\nタスクツリー + TaskRowItem"]
  Page --> Timeline["Presentationの期間射影"]
  Timeline --> Pane["読み取り専用タイムライン"]
  Pane --> Existing["既存タスク詳細"]
```

- 開始予定と期限の両方がある場合は期間バーにする。
- 片側だけの場合は、その日を表す1日バーにする。
- 両方がない場合は「日付未設定」へ分離する。
- 不正な日付は軸計算へ入れず、局所的に日付未設定相当として扱う。
- `timeline` をUI設定の最後のビュー許可値へ追加するが、粒度と表示基準日はセッション内のPresentation stateに限定する。

## 破綻シナリオ

- 無制限の日付列を生成すると、極端な日付を持つ1件だけでメモリとDOMが増大する。日14日、週12週、月12か月へ固定する。
- 期間専用Queryを先行導入すると、リスト、今日、お気に入りの条件とページ順が一覧から乖離する。初回は同じ `ListTaskPage` を正とする。
- タイムライン操作をカレンダー予定移動Use Caseへ接続すると、開始予定・期限・予定期間の意味が混ざる。初回は更新操作を提供しない。
- 不正な日付の差分計算が `NaN` になるとバーのCSSが壊れる。ISO日付を検証し、無効値を軸から除外する。
- 画面全体へ横スクロールを伝播すると左ペインと上部操作が見失われる。スクロール境界をタイムライン内部に限定する。

## スケール

- 1回の読み込み上限は既存どおり200件で、次ページは利用者操作で追記する。
- 行高と軸幅を固定し、動的なタイトルや日付でレイアウトを伸縮させない。
- ページ済みデータだけを並べ替え・射影し、全件を再取得しない。
- 将来、1ページ200件でも描画時間が基準を超えた場合は行仮想化を追加する。初回から専用Queryを増やす代替案は変更量に対して効果が不確実なため見送る。

## セキュリティと権限境界

- タイトルはReactのテキスト描画だけを使い、HTMLとして解釈しない。
- 日付範囲を自由入力させず、固定粒度と前後移動だけを許可する。
- 外部通信、ファイルアクセス、OS権限を追加しない。
- 読み取り専用表示からRepository更新、通知登録、タイマー副作用を呼ばない。
- ログへタスク名、メモ、通知本文を追加しない。

## トレードオフと代替案

- ページ外のタスクは追加読み込みまで見えない。一方で、一覧と同じ対象集合と性能境界を維持できる。
- 固定窓では長期計画全体を一画面に収められない。一方で、描画量と操作位置が予測可能になる。
- カレンダー週表示の再利用はコード量を抑えられるが、時刻配置と期間比較の責務が異なるため採用しない。
- 専用期間Queryは将来の集計や仮想化に有効だが、初回はフィルター重複を避けるため採用しない。
