# カレンダー時間グリッド表示回帰の設計レビュー

対象: GitHub #152 日・週表示の時間グリッドが消える回帰。

## 指摘事項

- CSS Gridでは明示配置された予定レイヤーも占有領域として扱われ、自動配置の時間セルが同じ行を避ける。
- 罫線色だけを変更しても、セル本体が表示位置から押し出されているため解決しない。
- 予定レイヤーを背面へ下げると、予定選択とリサイズハンドルの操作が失われる。
- スクリーンショットだけの確認では、日表示または将来のDOM順変更で再発を検出できない。

## 判断

承認。

- 時間ラベルと時間セルへ明示的なGrid行・列を割り当て、予定レイヤーとの重なりを意図した構造にする。
- 予定レイヤーの `z-index` と `pointer-events` は維持し、予定操作を変更しない。
- 日・週の両方でセル数、Grid座標、上端一致、罫線を自動検証する。
- DB、Read Model、Application Use Case、権限は変更しない。

## 破綻シナリオ

- 8:00ラベルと最初の時間セルが異なるGrid行に配置される。
- 週の一部日付だけセルが自動配置のまま残る。
- 日表示の単一列だけ列番号がずれる。
- 罫線は戻るが前面予定がセルに遮られ、クリックできない。
- 横スクロール後に時間ラベルと日列の行高がずれる。

## スケール

- 座標計算は固定15時間と表示日数1件または7件だけで、予定件数に依存しない。
- DOM要素数とRead Model取得範囲は変更しない。
- Grid座標の追加で再描画やTauri commandを増やさない。

## セキュリティ

- styleへ設定する値は内部で計算した整数だけとする。
- ユーザー文字列をHTML、style、data属性、ログへ追加しない。
- 外部通信、Tauri capability、OS権限、DB変更を追加しない。

## テスト不足と確認項目

- 日・週表示の全時間セル座標と罫線。
- 重複予定の選択、D&D、キーボード移動、期間端リサイズ。
- ドラッグ範囲作成と現在時刻線のヒット領域。
- 月表示の回帰なし。
- スモーク/標準大量データの警告件数。

## 代替案

時間セルの罫線を予定レイヤーへ疑似要素として描画する案。

不採用理由: 見た目だけは戻るが、時間セル自体のフォーカス、D&D、範囲作成座標が表示位置と一致しない。Gridセルの配置を正す必要がある。

## 残余リスク

- 日付ヘッダーまたは上部予定行の行数を変更した場合、時間行開始位置の更新が必要になる。座標検証をCIで継続し、構造変更時に検出する。
