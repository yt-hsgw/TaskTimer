# 左ペインのリスト管理メニュー 設計レビュー

対象: GitHub Issue #163 / `docs/issues/068-list-actions-overflow-menu.md`

## 指摘事項

- リスト行の主操作はビュー選択とし、編集・削除は件数右側の縦三点メニューへ分離する。
- ポップアップは左ペインのスクロール領域で切れないよう `document.body` へポータル表示する。
- 開いているメニューはリストIDを正として1件だけ保持し、別リストのメニューを開いた時に置き換える。
- 編集フォームと削除確認は既存Presentationを再利用し、更新・削除の検証とトランザクションは既存Application Use Caseに残す。
- 編集フォームは領域全体のフォーカスアウトを保存境界とし、内部の入力・色選択間の移動では保存しない。
- メニューを閉じる操作とフォーカス復帰を同じ関数へ集約し、`Escape` では元のトリガーへ戻す。

## 設計理由

左ペインは `overflow-y: auto` のため、行内の絶対配置では下端付近のポップアップが切れる。ポータルと固定座標を使い、トリガーの表示位置とポップアップ実寸から上下配置を決めることで、ペインのスクロールと狭いウィンドウの両方へ対応する。管理状態は永続化対象ではなく、ドメインモデルやRepositoryへ持ち込まない。

フォーカスアウト保存は入力ごとの即時更新と異なり、編集を終えた時にだけ既存Update Use Caseを1回呼ぶ。空白名称はPresentationで既存名称へ戻してから差分判定し、無効入力を保存境界へ渡さない。

## 代替案とトレードオフ

代替案は、編集フォームと同様に行の直下へ管理操作を展開する方式である。実装は単純だが、開閉で後続行が移動し、下端ではスクロール量も変わるため採用しない。ポータル方式は座標計算とフォーカス管理が増えるが、リスト行の高さを固定できる。

編集の代替案は、文字入力や色選択のたびに保存する方式である。明示保存は不要になるが、入力途中の値とDB更新回数が増え、失敗時の復元も複雑になるため採用しない。

## 破綻シナリオ

- メニューを開いたまま左ペインを閉じると孤立するため、`isOpen` の変更を監視して閉じる。
- 外側クリック後にトリガーへ強制フォーカスすると利用者の次操作を奪うため、フォーカス復帰は `Escape` とメニュー内の明示操作に限定する。
- 削除後に存在しないトリガーへフォーカスしないよう、DOM接続状態を確認する。
- 長いリスト名と大きい件数で末尾ボタンが押し出されないよう、行を `minmax(0, 1fr) auto` の固定トラックにする。
- スクロールやリサイズでアンカー位置が変わった場合はメニューを閉じ、画面上に取り残さない。
- 編集フォーム終了時はトリガーDOMが再生成されるため、フォーカス要求を保持し、再マウント後のレイアウト処理で適用する。単発のタイマーには依存しない。
- フォーカスアウト保存後に縦三点へ強制フォーカスするとクリック先の操作を奪うため、自動保存ではフォーカスを復帰しない。
- 更新失敗時に編集フォームを閉じると入力内容を失うため、Use Caseが成功した場合だけ編集状態を終了する。

## スケール懸念

全リストへ外側クリック監視を個別登録せず、`LeftNavigation` で1組だけ登録する。メニューDOMも開いている1件だけ生成するため、リスト件数に比例する追加イベントやポップアップ描画を発生させない。

## セキュリティと権限境界

- リスト名はReactのテキストとして描画し、HTMLへ変換しない。
- 名称と色の入力検証、既定リストの名称変更禁止、削除禁止は既存Application Use Caseを維持する。
- 削除前の確認を残し、所属タスクの既定リスト移動を既存トランザクションで行う。
- OS権限、外部通信、ログ出力は追加しない。

## テスト不足への対応

- 既定リストのメニューには編集だけ、カスタムリストには編集と削除が表示されること。
- メニューから編集フォームを開き、色を保存できること。
- 編集領域外へのフォーカス移動で保存され、移動先のフォーカスが維持されること。
- 空白のカスタムリスト名は更新せず、編集前の名称へ戻ること。
- 外側クリック、`Escape`、別ビュー選択、左ペイン折りたたみで閉じること。
- 長いリスト名と3桁件数でも行末の縦三点ボタンが表示されること。
- TypeScriptビルド、大規模データUI計測、README実画面キャプチャを通すこと。

## 判断
