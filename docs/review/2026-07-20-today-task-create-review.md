# 今日ビュー・かんばんのタスク作成レビュー

対象: 今日ビューから開始予定日付きタスクを追加し、かんばんの共通作成導線を回帰保証する。

## 指摘事項

- 今日から開始予定日だけを設定しても、従来のToday Read Modelは期限日しか見ないため、作成直後に一覧から消える。
- ページ取得、総件数、左ナビ件数、カレンダーでToday条件が分散しており、一部だけ変更すると表示と件数がずれる。
- 今日をDB上のリストとして作ると、実リストとの所属が二重になり既存モデルを壊す。
- 期限日も自動設定すると、意図しない期限通知が作成される。

## 判断

承認。

- 今日ビューは仮想スコープのままとし、最後に選択した実リストへ保存する。
- `TaskCreatePreset.standard` に開始予定日を追加し、今日スコープだけローカル日付の今日を設定する。
- Today Read Modelは親または未削除サブタスクの開始予定日・期限日を同じ条件で扱う。
- かんばんは既存の共通作成導線を維持し、ブラウザ回帰テストで保存まで保証する。

## 破綻シナリオ

- 今日から作成したタスクが作成後の再取得で消える。
- サブタスクだけが今日開始の場合に、親タスクが今日ビューへ表示されない。
- 今日ページの行数と左ナビの件数が一致しない。
- 今日スコープのカレンダーだけ開始予定タスクを欠落させる。
- 最後に選択した実リストが作成直前に削除され、保存に失敗する。

## スケール

- 新しい全件読み込みは追加しない。既存のページングクエリと日付範囲付きカレンダークエリ内で判定する。
- サブタスク判定は既存の相関 `EXISTS` を拡張し、結果行を増幅させない。
- `planned_start_date` と `due_date` は既存インデックス対象のため、スキーマ変更と追加マイグレーションは不要。

## セキュリティ

- SQLは既存のバインド変数だけを使い、日付やIDを文字列連結しない。
- 日付とタスク入力は既存Application Use Caseの検証を通す。
- タスク名、メモ、日付をログへ追加しない。
- 外部通信、Tauri capability、OS権限を追加しない。

## テスト

- Repository: 親の開始予定・期限、サブタスクの開始予定・期限をTodayページと件数で検証する。
- Calendar: 開始予定日だけが今日のタスクをTodayスコープで取得できることを検証する。
- Presentation: 今日からの作成要求が開始予定日=今日、期限なしになることを検証する。
- Presentation: かんばんの追加ボタンから共通ダイアログを保存できることを検証する。

## 残余リスク

- 日付をまたいでアプリを開き続けた場合、既存の今日更新契機に依存する。本変更では日付境界タイマーを追加しない。
- 実リストが作成直前に削除された場合は保存失敗として入力を保持し、別リストの自動選択は行わない。

## 検証結果

- Frontend: TypeScript型検査とVite本番ビルドに成功。
- Rust: 全108テスト成功。Clippyは全ターゲット・全機能を警告エラー扱いで成功。
- Presentation smoke: 50タスク・200サブタスクで今日/かんばん作成を含む全計測が成功し、WARN 0。
- README: 3画面のスクリーンショット生成と目視確認に成功。
- Privacy: 実行時外部通信、ログ、リモートアセット、更新機能権限の静的監査に成功。

## 代替案

今日から作成したタスクへ期限日も今日として設定する案。

不採用理由: 開始予定と期限通知の業務意味を混同し、依頼されていない通知を発生させるため。
