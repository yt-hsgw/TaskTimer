# 2026-07-20 ポモドーロ円形ゲージUI設計レビュー

## 対象

- Issue #141 で追加した独立ポモドーロ画面
- 未開始、作業中、休憩中、一時停止中の表示と操作配置

## 指摘事項

1. 現行UIは状態ごとに操作ボタン数が1、4、5個へ変わり、残り時間の中心位置と操作の探索順が安定しない。
2. 大きな矩形内へ情報を縦積みしているため、時間経過を直感的に把握できず、中央ペイン内のカードが二重に見える。
3. 作業と休憩の背景色差だけでは、色覚差や低コントラスト環境でフェーズを判別しにくい。

## 判断

フォローアップ付き承認とする。

- 残り時間を視覚タイマー型の円形残量ディスクとして描き、円の直下へフェーズ名、デジタル時間、完了セットを置く。
- 主操作を3スロットへ固定する。左は一時停止/再開、中央は開始または次フェーズ、右は終了とする。
- 状態固有の完了/スキップは高さ固定の補助操作領域へ置く。
- 作業/休憩は色とフェーズ名を併用し、円形ゲージは装飾ではなく時間進捗を表す。

## 設計理由

時間と進捗を同じ視線位置で読めるようにし、状態遷移後も操作位置を学習可能にする。主操作と例外的な補助操作を分離することで、利用頻度の高い開始、一時停止、次フェーズ、終了を探しやすくする。

## トレードオフ

- 円形残量ディスクは横長画面の余白を多く使うが、集中画面では情報密度より残り時間の視認性を優先する。
- 全面の残量表示では中央テキストとのコントラスト確保が必要なため、中央に固定色の読み取り領域を設ける。
- 補助操作が主操作より一段下がるため、作業だけを完了して停止する操作は1段階視線移動が増える。

## 代替案

横長の線形プログレスを残り時間の下へ置く案。

不採用理由: タスク一覧の進捗バーと見分けにくく、独立した集中機能としての視覚的な識別が弱い。

## 破綻シナリオ

- 残り0秒でstroke計算が負値または100%超過になる。
- 再開時に開始時刻が変わり、ゲージが逆方向へ跳ねる。
- 1分など短い設定で1秒更新のアニメーションが進捗値を追い越す。
- 長い日本語ラベルが狭幅で折り返し、操作列の高さが状態ごとに変わる。
- 通常タイマー実行中に開始ボタンだけ無効となり、理由が読み上げ利用者へ伝わらない。

## スケール懸念

ゲージ更新はポモドーロ画面内の1秒周期stateに限定し、アプリ全体や左ペインを再描画しない。SVG要素数は固定で、タスク件数に比例しない。

## セキュリティ・権限境界

- ユーザー入力やHTML描画を追加しない。
- Tauri Command、DB、OS通知権限を変更しない。
- 外部画像、Webフォント、外部通信を追加しない。

## テスト不足と確認方針

- 経過率0%、途中、100%でゲージ値を確認する。
- 未開始、作業、短休憩、長休憩、一時停止で固定スロットを確認する。
- 560px以下で時間と操作ラベルが重ならないことをスクリーンショットで確認する。
- 既存のUI性能計測で開始操作と画面表示の閾値を維持する。

## 実装後確認

- `npm run build`: 成功。
- `npm run audit:runtime-privacy`: 成功。外部通信、リモートアセット、実行時ログ、更新権限の追加なし。
- `npm run perf:ui`: 成功、WARN 0件。ポモドーロ表示33ms、開始28ms。
- DOM計測で未開始/作業中とも主要操作3スロット、補助操作2スロットを確認した。
- 開始前後で円形残量ディスク、主要操作、補助操作、主ボタンの座標と寸法が1pxを超えて変化しないことを確認した。
- 1440x900のスクリーンショットで残量ディスク、残り時間、操作、設定サマリーに重なりがないことを確認した。
