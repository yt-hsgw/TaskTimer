# 独立ポモドーロ設計レビュー

## 対象

- GitHub Issue #141
- `docs/issues/057-standalone-pomodoro.md`
- ポモドーロのドメイン、SQLite Repository、Tauri Command、専用画面、エクスポート

## 指摘事項

1. 現行モデルは作業フェーズにタスク対象と通常タイマー履歴を必須化しており、UIだけを独立させる変更はできない。
2. 対象列のnullable化だけでは矛盾した行を作成できるため、scopeと対象・タイマーの組み合わせをCHECK制約で固定する必要がある。
3. 通常タイマーとの排他を画面状態だけに任せると、多重クリックや再起動直後の競合で同時実行が発生する。
4. 旧形式の進行中セッションを読み込めなくすると、更新直後に停止・完了できなくなる。

## 破綻シナリオ

- マイグレーション途中で旧テーブルを削除し、コピー失敗によって履歴を失う。
- 独立作業フェーズが通常タイマー履歴を作り、タスク別実績へ誤集計される。
- 自動休憩後の次作業開始だけが旧タスク連携ロジックを通り、対象なしで失敗する。
- タスク削除がNULL対象の独立セッションまで巻き込む。
- OSスリープ後に画面上だけ0秒となり、DB状態と通知が確定しない。

## スケール

- アクティブ取得は部分ユニークインデックスを維持し、履歴全件を読み込まない。
- nullable対象の導入後も対象別エクスポート・削除用インデックスを維持する。
- 1秒ごとのカウントダウンは専用パネル内部だけを更新し、App全体のread modelを再取得しない。

## セキュリティと権限境界

- 新しい入力値、外部通信、OS権限は追加しない。
- DB制約とRepositoryの状態検証を権限境界とし、Tauri Commandから任意scopeや対象IDを受け取らない。
- 通知文へタスク内容を含める既存設定は旧形式だけに限定し、独立セッションは汎用タイトルを使う。
- マイグレーションとエクスポートでタスク名・メモをログへ出さない。

## テスト不足と対応

- 旧スキーマ移行、scope整合制約、通常タイマーとの排他を追加する。
- 独立scopeの全状態遷移と自動遷移を追加する。
- JSON/CSV互換と既存タスク連携履歴の保持を追加する。
- 専用画面の基本操作をUI計測へ追加する。

## 代替案

内部の「未分類」タスクへ自動紐付けする案はDB変更が小さい。一方で非表示タスク、意図しない通常タイマー履歴、削除・集計の例外が増えるため不採用とする。

## 判断

フォローアップ付き承認。旧形式を読み書き可能な互換経路として残し、新規開始Commandからscopeや対象を受け取らないことを条件に実装へ進む。
