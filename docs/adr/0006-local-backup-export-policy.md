# ADR 0006: ローカルバックアップとエクスポート方針

## 状態

採用。

## 文脈

TaskTimerはローカルSQLiteを唯一の正とする。実務利用では端末故障、端末移行、誤削除に備え、何を退避すればよいか、どの形式を復元用とするか、破損やバージョン不一致時にどう扱うかを明確にする必要がある。

アプリ実行時の外部通信禁止、クラウド同期なし、自動更新なしの方針は維持する。

## 決定

- 完全復元用の正はSQLiteバックアップにする。
- SQLiteバックアップは `tasktimer.sqlite3` と `backup-manifest.json` の組み合わせを標準形にする。
- JSON/CSVは閲覧、監査、他ツール移行の補助形式として扱い、完全復元形式にはしない。
- 復元時は一時DBで `PRAGMA integrity_check`、必須テーブル、マイグレーション適用可否を確認してから入れ替える。
- 新しいアプリで作られたDBなど、現在のアプリが理解できないDBは復元を拒否する。
- バックアップ/エクスポートファイルは個人データとして扱い、Issue、PR、Discussions、Release artifactへ添付しない。
- 実装は次のIssueへ分割する。
  - GitHub #88: SQLiteバックアップ/復元Use Case。
  - GitHub #87: JSON/CSVエクスポートUse Case。
  - GitHub #89: 設定画面のデータ管理UI。

詳細は [ローカルデータのバックアップとエクスポート方針](../data-backup-export.md) に記録する。

## 設計理由

- SQLiteバックアップは、タスク、サブタスク、通知ルール、タイマー履歴、UI設定を一貫して退避できる。
- JSON/CSVを完全復元形式にすると、SQLite制約、削除済み行、通知ルール、将来マイグレーションを別形式で再実装する必要がある。
- manifestを同梱すると、復元時にアプリバージョン、作成日時、整合性確認結果を説明できる。
- バックアップ/エクスポートを設定画面に置くと、通常作業ビューを邪魔せず、危険操作として注意喚起しやすい。

## トレードオフ

- SQLiteバックアップは完全復元に向く一方、人間が内容を確認しづらい。
- JSON/CSVは確認しやすい一方、完全復元には使えない。
- アプリ内バックアップ機能を追加するとファイル選択とローカルファイル権限の設計が必要になる。
- 暗号化バックアップを見送るため、利用者はバックアップファイルの保管先を自分で守る必要がある。

## 代替案

OS標準バックアップへ完全に任せる。

不採用理由:

- 利用者に退避対象が伝わらない。
- 復元時の破損DB、バージョン不一致、マイグレーション失敗をTaskTimerとして説明できない。

JSONを完全復元形式にする。

不採用理由:

- 完全復元に必要な制約と互換性管理がSQLiteより複雑になる。
- タイマー履歴、通知ルール、削除済み行を正確に戻す責務が重い。

## セキュリティ

- SQLiteバックアップ、JSON、CSVにはタスク名、サブタスク名、メモ本文、タイマー履歴、通知ルールが含まれる可能性がある。
- Issue、PR、Discussions、Release artifactへバックアップ/エクスポートファイルを添付しない。
- エラー表示とログにはユーザー内容を含めない。
- クラウド保存、自動バックアップ、暗号化、パスワード管理はMVP外とし、必要になった時点で別ADRで扱う。

## 破綻シナリオ

- DBコピー中の書き込みにより、破損したバックアップが作られる。
- 復元失敗時に既存DBまで失われる。
- 新しいDBを古いアプリへ復元して起動不能になる。
- エクスポートファイルを公開Issueへ添付し、個人情報が漏れる。
- CSVのエスケープ不備でメモ本文が壊れる。

## 受け入れ条件

- バックアップ対象と対象外がdocsで明確である。
- SQLite、JSON、CSVの役割が分かれている。
- 復元時の破損DBとバージョン不一致の扱いが定義されている。
- 後続実装IssueがUse Case、Repository/Infrastructure、UIへ分割されている。
