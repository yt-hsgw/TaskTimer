# 054 かんばんカード全面のドラッグ操作と前面表示を改善する

GitHub Issue: #138

## 背景

現在はカード左端の小さなハンドルだけがドラッグ開始領域であり、状態変更の操作負荷が高い。
また、ドラッグ中のカードを元の状態列内で変形しているため、別列へ移動すると列のスクロール領域に切り取られ、変更先カードの背面へ入ったように見える。

## 要件

- タスクカードの任意の位置からポインターでドラッグを開始できる。
- 完了チェックと詳細表示は、移動量がドラッグ開始閾値未満なら従来どおりクリックとして扱う。
- キーボードでのカード移動を維持する。
- ドラッグ中のカードは状態列のスクロール領域に切り取られず、全列より前面に表示する。
- ドロップ先列の強調表示と保存失敗時のDB再取得を維持する。
- ドロップ解放から保存済みRead Model取得まで、移動元カードを再表示しない。

## UI状態

- `PointerSensor` の6px開始閾値を維持し、単純クリックとドラッグを分ける。
- `DragOverlay` に移動中カードのプレビューを描画する。
- 元カードはドラッグ中だけ薄くし、同じカードが2件存在するように見えないようにする。
- カード内の専用ドラッグハンドルは廃止し、完了チェックと本文の2列構成へ整理する。
- ドラッグ開始中の一時状態はPresentation内に留め、永続化しない。
- ドロップ後は移動先列へ楽観表示し、保存失敗時だけ移動元へ戻す。保存成功時はRead Modelの列IDを確認して楽観状態を解除する。

## トランザクション境界

変更しない。ドロップ完了時だけ既存 `MoveTaskToBoardColumn` Use Caseを呼び、`board_column_id` と互換状態を1トランザクションで更新する。

## 設計理由

CSSの `z-index` だけでは、祖先要素の `overflow-y: auto` による切り取りを越えられない。D&Dライブラリのオーバーレイを利用し、ドラッグ中の見た目を列DOMから分離する。

## トレードオフ

- カード全面をドラッグ起点にすると、テキスト選択より状態移動を優先する。
- オーバーレイ用にカード表示を再利用する必要があり、表示項目追加時は通常カードとの同期が必要になる。

## 代替案

カードの `z-index` だけを上げる。

不採用理由: 列のスクロール領域によるクリッピングを解消できず、列をまたいだ前面表示を保証できない。

## セキュリティと危険ケース

- ドロップ先列IDは既存Application層で存在確認する。
- タイトル、メモ、タグは文字列として描画し、HTMLへ変換しない。
- 外部通信、OS権限、Tauri capabilityを追加しない。
- 完了ボタンのクリックがドラッグとして誤認される。
- ドラッグ終了時に対象列が削除済みになっている。
- 大量カードでオーバーレイ生成により描画が重くなる。
- 保存待ち中に移動元カードが一瞬再表示される。

## 受け入れ条件

- カード本文、余白、チェック付近からポインター移動でドラッグできる。
- 単純クリックでは完了切替または詳細表示が実行される。
- 別列へ移動中のカードが変更先カードの前面に表示される。
- ドロップ解放直後の最初の描画フレームから、カードが移動先だけに表示される。
- キーボードD&Dと保存失敗時の復元が維持される。
- Frontendの型検査と本番ビルドが成功する。

## 実装結果

- カード内の専用ハンドルを廃止し、カード全面へPointer/Keyboard D&Dリスナーを移した。
- `DragOverlay` を `document.body` 直下へ描画し、状態列のスクロール領域による切り取りを回避した。
- 列の空白だけでなく列内カード上でも、移動先の状態列全体を強調する。
- 保存待ち中は移動先へ楽観表示し、Read Model更新後に楽観状態を解除することで移動元のちらつきを防止した。
- 既存の大量データUIスモークへ、クリック、カード本文ドラッグ、前面表示、列強調、ドロップ直後の配置、保存後移動の回帰検証を追加した。
