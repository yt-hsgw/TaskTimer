# 011: 右詳細ペインを実装する

GitHub Issue: #29

## 目的

タスク選択時だけ右ペインを開き、サブタスク、期限、通知、タイマー、削除を一か所で操作できるようにする。

## スコープ

- タスク詳細表示。
- サブタスク追加。
- サブタスク期限設定。
- タスク期限設定。
- 通知設定。
- タイマー目標時間設定。
- タイマー開始/終了。
- タスク削除。

## スコープ外

- タイマー一時停止/再開。
- 繰り返し設定の永続化。
- ファイル添付。
- OS側の将来時刻通知予約。既存MVPと同じく、期限到来通知はアプリ起動中または再読み込み時のdispatch対象として扱う。

## 実装方針

- 中央ペインのタスク一覧と右ペインのタスク詳細をPresentation上で分離する。
- 右ペインはタスク選択時だけ表示し、初期表示では自動選択しない。
- 右ペインではタスクのタイトル、開始日、終了日、メモ、タイマー目標時間を更新できる。
- サブタスクもタイトル、開始日、終了日、メモ、タイマー目標時間を更新できる。
- 通知設定は開始日/終了日に紐づく通知予定として扱う。開始日または終了日が設定された場合は通知ルールを有効化し、日付が削除された場合は対象の通知ルールを無効化する。
- タイマー操作は既存の開始/終了Use Caseを使う。一時停止/再開は #30 で扱う。
- 削除は既存のソフト削除Use Caseを使い、確認メッセージを表示してから実行する。

## 設計レビュー

### データモデル

既存タスク/サブタスク/通知/タイマーに加え、タイマー目標時間が必要。`timer_target_seconds` は既存スキーマに存在するため、追加マイグレーションは不要。

### トランザクション境界

- タスク詳細更新は、入力検証、タスク更新、通知ルール同期を同一トランザクションで行う。
- サブタスク詳細更新は、入力検証、サブタスク更新、通知ルール同期を同一トランザクションで行う。
- 削除は既存のソフト削除トランザクションを維持する。
- タイマー開始/終了は単一アクティブタイマー制約を維持する。

### セキュリティ

メモ本文をHTMLとして描画しない。通知本文にメモを含めない。
タイトル、日付、メモ、タイマー目標時間はApplication Use Caseで検証する。アプリ実行時の外部通信は追加しない。

### 破綻シナリオ

- 日付更新時に古い通知ルールが残ると、意図しない日時に通知される。
- メモだけ更新した場合に通知ルールを再作成すると、過去通知が再送対象になり得る。
- 右ペインを初期表示で自動的に開くと、ユーザーがまだ選択していないタスクを編集対象と誤認する。
- サブタスクタイマー開始中に親タスクを削除する場合は、既存のソフト削除処理で開始中タイマーも通常検索から除外する必要がある。

### スケール

- 右ペインは選択タスクの既存詳細データだけを使い、全タイマー履歴を読み込まない。
- サブタスク編集フォームは選択タスク配下だけに限定し、一覧全体へフォーム状態を持たせない。

## トレードオフ

- 詳細ペインへ操作を集約すると便利だが、狭い幅では誤操作を避ける余白と確認が必要。
- サブタスク編集を同一ペイン内で行うと画面遷移は少ないが、状態管理が複雑になる。
- 通知設定を独立した有効/無効トグルではなく日付設定に紐づけるとMVPは単純になるが、日付は保持して通知だけ止める操作は後続Issueになる。

## 代替案

詳細をモーダルで開く。

不採用理由:

- タスク一覧を見ながら作業しづらい。

## 受け入れ条件

- タスク選択時だけ右ペインが開く。
- 右ペインでサブタスク追加と期限設定ができる。
- 右ペインで通知表示とタイマー操作ができる。
- 削除時は確認が表示される。

## レビュー判断

フォローアップ付き承認。

- #29では右詳細ペインと既存データモデルで表現できる更新操作を優先する。
- 繰り返し、一時停止/再開、通知の独立トグルは #30 以降で扱う。
