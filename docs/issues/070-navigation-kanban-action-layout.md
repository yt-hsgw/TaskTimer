# 070 左ペインとカンバンの操作配置を整理する

GitHub Issue: #178

## 背景

左ペインではスマート範囲よりリスト管理が先に表示され、日常的に使う「今日」「お気に入り」が見つけにくい。かんばんではタスク追加と状態追加が画面上部へ集まり、操作対象となる状態列との対応が分かりにくい。

## 仕様

- 左ペインは `今日`、`お気に入り`、`リスト` の順に表示する。
- リストセクションは展開・折りたたみ可能とし、初期状態は展開とする。
- リスト追加は一覧の下部に置く。入力中は同じ位置に作成フォームを表示する。
- リストセクションを閉じると、編集中フォームと縦三点メニューも閉じる。
- 左ペイン全体をアイコン表示へ縮小した場合は、既存のリスト移動導線を維持する。
- かんばんの状態追加は、状態列の最右端にある追加スロットから開始する。
- 各状態列の未完了一覧上部に `タスクを追加` を表示する。
- 状態列から作成したタスクは、その状態列へ同一トランザクションで所属させる。
- 状態列の並べ替えハンドルは列カード最上部中央へ移し、名称編集や削除とは分離する。
- 状態列の並べ替えハンドルは横バーの見た目を維持し、列幅全体をPointer操作領域にする。
- 状態列見出し右端は削除ボタンではなく縦三点メニューとし、状態名編集、状態削除、完了タスクの全件削除、列内タスクの並べ替えを集約する。
- 完了タスクの全件削除は確認後に実行し、読み込み済みページではなくDB上でその状態列に所属する完了タスクを対象にする。
- 列内タスクの並べ替えは `既定順`、`期限が近い順`、`作成日の新しい順`、`タイトル順` から列ごとに選ぶ。表示順だけを変更し、永続データの `sort_order` は更新しない。
- 追加スロットは状態列の並べ替え対象にも、タスクのドロップ先にも含めない。

## データとトランザクション境界

新しいテーブルや列は追加しない。既存の `tasks.board_column_id` を利用する。

カンバン列からの作成はApplication Use Case `CreateTaskInBoardColumn` を入口にし、次を1トランザクションで行う。

1. タスク入力と状態列IDを検証する。
2. 状態列が削除されていないことを確認する。
3. タスク作成時に `board_column_id` を指定する。

通常の一覧・今日・ショートカットからの作成は既存 `CreateTask` を維持し、既定の有効状態列へ保存する。

完了タスクの全件削除はApplication Use Case `DeleteCompletedTasksInBoardColumn` を入口にし、状態列IDを検証して、対象列の `lifecycle_status = done` である未削除タスクとそのサブタスク、タイマー履歴、ポモドーロ履歴、通知ルール、繰り返し、タグ関連を1トランザクションでソフト削除する。未完了タスクと他の状態列は変更しない。

## 設計理由

- スマート範囲を上に置くと、期限やお気に入りを起点にする毎日の確認が短い移動で済む。
- 操作を対象の近くへ置くと、どの状態へタスクを追加するか、どこへ状態を追加するかが明確になる。
- 作成後に `MoveTaskToBoardColumn` を呼ぶ二段階方式では、移動失敗時にタスクだけが別列へ残るため、専用Use Caseで原子的に保存する。
- リスト折りたたみは作業中だけの表示状態とし、UI設定スキーマとDBマイグレーションは増やさない。
- 列メニューへ管理操作を集約すると、列見出しの横位置を安定させたまま破壊的操作を確認付きで提供できる。
- ソートはPresentationの派生表示に限定し、タスク移動や既存 `sort_order` の意味を変更しない。

## トレードオフ

- 各状態列に追加操作が1つ増えるが、画面上部の曖昧な追加ボタンをなくせる。
- カンバン用作成コマンドが増えるが、トランザクション境界が明確になり、部分成功を防げる。
- 折りたたみ状態は再起動後に展開へ戻る。永続化よりも設定モデルを小さく保つことを優先する。
- 列ごとのソート選択は再起動後に既定順へ戻る。永続化する場合はUI設定の列ID別マップと削除列の掃除が必要になるため、本Issueでは一時状態とする。
- 完了タスクの全件削除は対象数に比例してタスクグラフ削除を行うが、単一トランザクションにまとめて部分成功を防ぐ。

## 代替案

画面上部の追加ボタンを残し、ダイアログ内で状態列を選択する案。

不採用理由:

- タスクを追加するたびに列を選ぶ操作が増える。
- 操作の起点と保存先が離れ、カンバン上での直接操作という期待に合わない。

## 危険ケース

- 状態追加スロットがD&Dの並べ替え対象になり、状態順の保存値が壊れる。
- 状態列からタスクを作成中にその列が削除され、存在しない列IDで保存する。
- リストセクションを閉じてもポータルメニューが残る。
- 列ハンドルのPointer操作が名称編集やカードD&Dへ伝播する。
- カンバン作成と列移動を別トランザクションにし、移動だけ失敗する。
- ページ外の完了タスクが削除されず、件数表示とDBが食い違う。
- 完了タスクの一括削除中に一部だけコミットされる。
- 列メニューを開いたまま列D&Dを開始し、ポップアップが移動元へ残る。

## セキュリティと権限

- タスク名、メモ、リスト名、状態名は既存のApplication検証を通す。
- 状態列IDは識別子検証とRepositoryの存在確認を通す。
- ユーザー文字列をHTMLとして描画せず、ログにも出さない。
- 外部通信、OS権限、Tauri capabilityは追加しない。

## 受け入れ条件

- 今日とお気に入りがリストより上に表示される。
- リストを折りたたみ、再展開できる。
- リスト追加がリスト一覧の下部から行える。
- 状態追加を最右端の追加スロットから行える。
- 各状態列の先頭からタスク作成ダイアログを開ける。
- 作成したタスクが選択した状態列へ保存される。
- 状態列を最上部ハンドルからPointer/Keyboardで並べ替えられる。
- 状態列の横幅全体が並べ替えハンドルの操作領域になる。
- 縦三点メニューから状態名編集、状態削除、完了タスク全件削除、列内ソートを選択できる。
- 完了タスク全件削除は対象列の完了タスクだけをソフト削除し、未完了タスクと他列を維持する。
- 列内ソートを切り替えてもDB更新を行わない。
- 既存のカードD&D、完了チェック、状態名編集、列削除に回帰がない。

## 実装・検証結果

- Application/Repositoryへ `DeleteCompletedTasksInBoardColumn` を追加し、対象列の完了タスクグラフを1トランザクションでソフト削除するようにした。
- 正常系に加え、2件目の削除を強制失敗させても1件目が残るロールバックテストを追加した。
- Rust全テスト、Clippy、TypeScript/Viteビルド、実行時プライバシー監査、リリーススクリプトテストを通過した。
- 大量データUIスモークで列メニュー経由の状態名編集、列・カードD&D、他ビューの回帰がないことを確認した。
- README用のデスクトップ画像で、三点メニューの前面表示、項目の文字切れ、列内全幅ハンドルを確認した。
