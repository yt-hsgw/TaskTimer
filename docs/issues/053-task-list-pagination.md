# 053: タスク一覧の200件上限をページングで解消する

GitHub Issue: #131

## 背景

`list_tasks` と `list_task_rows` は1回200件を上限とするが、追加読み込み境界がない。大量DBでも初期描画は抑えられる一方、201件目以降へ到達できない。

## 要件

- タスクツリーとタスク行Read Modelへ安定したカーソルページングを追加する。
- タスク、今日、お気に入り、タグ、かんばんのスコープをDB側で絞る。
- 追加読み込み時に選択、展開、スクロール位置を維持する。
- DB集計件数と読み込み済み件数を区別する。

## 設計方針

カーソルは完了区分、`sort_order`、`created_at`、`id` の組をApplication DTOで検証する。Infrastructureは同じ順序で次ページを取得する。状態変更Use Caseのトランザクション境界と本番テーブルは変更しない。

## 代替案

仮想リストだけを追加する。

不採用理由: DOM数は減るが、Repositoryが返さない201件目以降へ到達できない問題は解消しない。

## 危険ケース

- 同順位タスクがページ間で重複または欠落する。
- ページ読み込み中の完了・削除でカーソルがずれる。
- 今日やお気に入りをPresentationの全件走査で実装する。
- タスク行と詳細用親子データが不整合になる。

## 受け入れ条件

- 5,000件DBで201件目以降へ到達できる。
- ページ間の重複・欠落がないRepositoryテストがある。
- 主要スコープで対象外データを全件取得しない。
- 外部通信、新しい権限、ユーザーデータのログ出力を追加しない。
