# 013: タイマー一時停止/再開と繰り返し設定を設計・実装する

GitHub Issue: #30

## 目的

右詳細ペインで要求されるタイマー一時停止/再開、終了、目標時間、繰り返し設定をデータモデルから実装する。

## スコープ

- タイマー一時停止/再開のドメインモデル。
- タイマー終了時の経過時間計算。
- タイマー目標時間。
- 繰り返し設定。
- 通知ルールとの整合。

## スコープ外

- UI Shell実装。
- 月カレンダー。
- 外部カレンダー同期。
- 完了時の次回タスク自動生成。

## 実装方針

- 一時停止区間は `timer_pauses` として `timer_sessions` に紐づける。
- 一時停止中も `timer_sessions.stopped_at IS NULL` のままにし、単一アクティブタイマー制約を維持する。
- 一時停止中は別タスク/サブタスクを開始できない。
- 終了時に未再開の一時停止区間がある場合は、終了時刻で `resumed_at` を閉じてから経過秒数を計算する。
- 経過秒数は `stopped_at - started_at - pause合計秒数` とする。
- 繰り返し設定は `recurrence_rules` としてタスク/サブタスクに紐づける。
- MVPの繰り返し頻度は `daily`, `weekly`, `monthly` とし、間隔は1以上365以下に制限する。
- 繰り返しを有効にする場合は、開始日または終了日の少なくとも一方を必須にする。これにより通知予定の基準日がない繰り返し設定を避ける。
- 繰り返し設定の保存、タスク/サブタスク日付更新、通知ルール同期は同一トランザクションで処理する。

## 設計レビュー

### データモデル

一時停止区間は `timer_pauses` で表現する。繰り返しは `recurrence_rules` としてタスク/サブタスクに紐づける。

### トランザクション境界

- 一時停止、再開、終了は単一アクティブタイマー制約を維持し、同一トランザクションで処理する。
- 繰り返し設定更新と通知ルール更新は同一トランザクションで扱う。

### セキュリティ

繰り返しや通知設定にメモ本文を含めない。
アプリ実行時の外部通信は追加しない。繰り返し頻度と間隔はApplication Use Caseで検証する。

### 破綻シナリオ

- 一時停止中に別タスクを開始できると、単一アクティブタイマー制約が破綻する。
- 未再開の一時停止区間を終了時に閉じないと、経過秒数が不定になる。
- 一時停止時間を経過秒数へ含めると、実作業時間の記録として不正確になる。
- 繰り返し設定だけ保存して開始日/終了日がない場合、通知予定とカレンダー表示の基準日がなくなる。
- 完了時に次回タスクを自動生成すると、重複生成や無限生成の危険がある。

### スケール

- 終了時のpause合計は対象タイマー1件のpause行だけを集計する。
- active timer取得では開いているpause 1件だけをJOINし、全タイマー履歴をUIへ渡さない。

## トレードオフ

- 一時停止/再開は実作業に合うが、経過時間計算とアプリ終了時復元が複雑になる。
- 繰り返しタスクは便利だが、完了時の次回生成で重複や無限生成の危険がある。
- 繰り返し有効時に日付を必須にすると入力制約は増えるが、通知ルールとカレンダー表示の基準を保てる。

## 代替案

タイマーは開始/終了だけに留め、UIでは一時停止を提供しない。

不採用理由:

- ユーザー要望の右ペイン操作に含まれており、実務タイマーとして不足する。

繰り返し設定と次回タスク生成を同時に実装する。

不採用理由:

- 次回生成は完了処理、通知再作成、重複防止の設計が必要で、#30の受け入れ条件より大きい。

## 受け入れ条件

- タイマーを一時停止できる。
- 一時停止中は別タスクのタイマーを開始できない。
- タイマーを再開できる。
- 終了時の経過時間から一時停止時間が除外される。
- 繰り返し設定が保存できる。
- 繰り返し設定は通知ルールと矛盾しない。
