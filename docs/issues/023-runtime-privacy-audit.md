# 023: 実行時外部通信・ログ出力の静的監査を追加する

GitHub Issue: #49

## 目的

TaskTimerの公開版がオフライン前提を維持していることを、Release前ゲートで確認しやすくする。
Windows実機での通信監視は引き続き #49 の手動確認として残るが、リポジトリ上では実行時コード、Tauri設定、権限、依存関係の静的監査を自動化する。

## スコープ

- `npm run audit:runtime-privacy` を追加する。
- 実行時コードから外部通信API、リモートURL、実行時ログ出力、CSS importを検出する。
- Tauri CSP、Capability、macOS entitlement、Updater artifact設定を確認する。
- npm/Cargoの直接依存に、実行時通信を導入しやすい依存がないか確認する。
- `リポジトリチェック` workflowへ監査を組み込む。
- セキュリティ資料とReleaseチェックリストへ監査コマンドを追記する。

## スコープ外

- Windows実機またはVMでのパケット監視。
- 公開Release artifactの実行時ログ採取。
- ネットワーク監視ツールの恒久導入。
- 外部通信、自動更新、クラッシュアップロード機能の追加。

## 設計レビュー

### データモデル

業務データ、SQLiteスキーマ、Repository境界には影響しない。
監査対象はソース、設定ファイル、依存関係定義である。

### トランザクション境界

アプリ実行時のトランザクションは変更しない。
監査スクリプトは読み取り専用で、失敗時に非0終了する。

### セキュリティ

スクリプトはタスク名、メモ本文、DBファイル、ログファイルを読み込まない。
実行時コードに `console.*`、`println!`、`dbg!` などのログ出力入口が入った場合に失敗する。
開発・運用用途の通信、たとえばGitHub Actions、Dependabot、READMEスクリーンショット生成用ローカルVite/Chrome接続は対象外とする。

### 破綻シナリオ

- 開発用スクリプトの通信を誤検知し、実行時通信と混同する。
- 依存関係が内部で通信するが、直接依存名の監査だけでは検出できない。
- Tauri pluginやcapabilityを追加したのに、監査対象パターンに含まれず見逃す。
- 手動実機確認が不要になったと誤解される。

### スケール

監査対象は実行時ソースと設定ファイルだけに限定する。
大きな生成スキーマやlock fileを走査しないため、CIで軽量に実行できる。

## トレードオフ

- 静的監査は速く再現性があるが、実際のOS/ネットワーク挙動を完全には保証しない。
- 厳しめに `console.*` や `println!` を禁止すると、将来の診断ログ追加時にPRで明示レビューが必要になる。
- 外部通信系依存の直接追加は検出できるが、間接依存の挙動はDependabotや手動レビューと併用する必要がある。

## 代替案

Windows実機でのパケット監視だけをRelease前に手動実施する。

不採用理由:

- 開発中のPRで外部通信入口が混入しても、Release直前まで検出できない。
- 実機確認環境がない期間に #49 の前段作業を進められない。

## 受け入れ条件

- `npm run audit:runtime-privacy` が成功する。
- CIの `リポジトリチェック` で同監査が実行される。
- 実行時外部通信なし、実行時ログ出力なし、リモートアセットなし、自動更新なしの確認観点が資料に残っている。
- Windows実機での通信監視が引き続き #49 の残作業として明示されている。

## レビュー判断

フォローアップ付き承認。

- 静的監査はRelease前ゲートへ追加する。
- 公開Release artifact上の実機確認は、Windows環境が用意できた時点で #49 に結果を記録する。
