# 026: 通知ルールの個別有効・無効UIを追加する

GitHub Issue: #55

## 目的

タスクまたはサブタスクの日付は残したまま、開始日通知・終了日通知だけを個別に停止または再開できるようにする。

## スコープ

- タスク詳細の通知セクションで、開始日通知と終了日通知のON/OFFを切り替える。
- サブタスク詳細でも開始日通知と終了日通知のON/OFFを切り替える。
- 通知ルールの `enabled` と `registration_status` をDBへ保存する。
- OFFの通知ルールを期限到来通知dispatch対象から除外する。
- 再ONにした通知ルールを `pending` に戻し、期限到来済みなら再dispatch対象にする。

## スコープ外

- OS側の将来時刻スケジューリング。
- 通知時刻の任意設定。
- 通知失敗履歴の一覧表示。これは #53 で扱う。

## 実装方針

- 既存 `notification_rules.enabled` を正とし、新規マイグレーションは追加しない。
- タスク/サブタスクRead Modelへアクティブな通知ルールを含める。
- `SetNotificationRuleEnabled` Use Caseを追加し、通知ルールIDを検証してRepositoryへ渡す。
- OFF操作はソフト削除ではなく `enabled = 0` と `registration_status = 'disabled'` にする。
- ON操作は `enabled = 1` と `registration_status = 'pending'` に戻し、`last_error` を消す。
- 日付削除時は既存方針通り通知ルールをソフト削除する。日付が残っている個別OFFとは状態を分ける。

## 設計レビュー

### データモデル

`notification_rules` はすでに `enabled`、`registration_status`、`deleted_at` を持つ。個別OFFは「通知意図は残るが送信対象外」、日付削除やタスク削除は「通知意図そのものをソフト削除」として扱う。

### トランザクション境界

- `SetNotificationRuleEnabled` は通知ルール1件の存在確認と状態更新を単一トランザクションで行う。
- OS通知送信は発生させない。送信は既存の `DispatchDueNotifications` のDBコミット後副作用に限定する。

### セキュリティ

通知トグル操作ではタスク名、サブタスク名、メモ本文、通知本文をログへ出さない。外部通信、追加権限、HTML描画は追加しない。

### 破綻シナリオ

- 日付を残して通知OFFにした後、タイトルだけ保存すると通知が勝手にONへ戻る。
- 再ONしても `registration_status` が `disabled` のままでdispatchされない。
- ソフト削除済み通知ルールをUIからONにできてしまう。
- サブタスクの通知状態が親タスクの通知状態と混同される。

### スケール

タスク一覧取得上限は既存の200件で、通知ルールは対象ごとに開始日・終了日の最大2件。Read Modelへの追加はタスク/サブタスク数に対して線形で、タイマー履歴の全読み込みは発生しない。

## トレードオフ

- タスクRead Modelへ通知ルールを含めると一覧取得のデータ量は増えるが、詳細ペイン表示時に追加クエリを重ねず状態を一貫して扱える。
- OFFをソフト削除にしないことで再ONが簡単になる一方、日付削除と個別OFFの違いをUse Caseとテストで明確にする必要がある。

## 代替案

詳細ペインを開いた時だけ通知ルールを別コマンドで取得する。

不採用理由:

- 右ペイン内のタスク/サブタスク編集と通知状態の同期が分かれ、保存直後の表示整合性が複雑になる。

## 受け入れ条件

- タスク通知の開始日/終了日を個別にOFF/ONでき、状態が保存される。
- サブタスク通知の開始日/終了日を個別にOFF/ONでき、状態が保存される。
- OFFの通知ルールはdispatchされない。
- 再ONした通知ルールは、条件を満たす場合にdispatch対象へ戻る。
- メモ本文や通知本文をログへ出さない。

## レビュー判断

フォローアップ付き承認。

- 通知時刻の任意設定と失敗履歴の一覧は別Issueで扱う。
- OSごとの実通知挙動は既存のWindows優先運用に合わせ、実機確認で継続確認する。
