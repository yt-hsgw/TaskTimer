# 026: 通知の全体有効・無効設定を追加する

GitHub Issue: #55

Status: 完了。通知失敗履歴の一覧改善は #53、OSへの将来時刻スケジューリングは #51 で継続する。

## 目的

設定画面で、TaskTimer全体の期限到来通知をON/OFFできるようにする。

## スコープ

- 設定画面で通知全体の有効/無効を切り替える。
- 全体OFF中は期限到来通知dispatchを実行してもOS通知を送信しない。
- 全体OFF中も、タスク/サブタスクの日付と通知ルールは保持する。
- 全体ONへ戻した場合、既存の `pending` / `failed` 通知ルールは通常どおりdispatch対象に戻る。

## スコープ外

- タスク/サブタスク詳細での通知個別ON/OFF。
- 開始日通知と終了日通知の個別ON/OFF。
- 通知時刻の任意設定。
- OS側の将来時刻スケジューリング。
- 通知失敗履歴の一覧表示。これは #53 で扱う。

## 実装方針

- `notification_preferences.notifications_enabled` を追加し、全体ON/OFFの正とする。
- 既存DBにはマイグレーションで `notifications_enabled = 1` を追加する。
- `UpdateNotificationsEnabled` Use Caseで設定を保存する。
- `DispatchDueNotifications` は全体OFFの場合、通知ルールを変更せず `attempted = 0` で終了する。
- 右詳細ペインの通知セクションは読み取り表示に留める。

## 設計レビュー

### データモデル

`notification_rules` は個別通知意図を表し、日付更新や削除と同期する。全体ON/OFFは `notification_preferences` に置き、通知ルールの `enabled` や `registration_status` は変更しない。

### トランザクション境界

- `UpdateNotificationsEnabled` は `notification_preferences` 1行の更新を単一トランザクションで行う。
- `DispatchDueNotifications` は全体OFF時にOS通知副作用を発生させない。

### セキュリティ

全体ON/OFF操作ではタスク名、サブタスク名、メモ本文、通知本文をログへ出さない。外部通信、追加権限、HTML描画は追加しない。

### 破綻シナリオ

- 全体OFFにしたのに、再フォーカスや再試行でOS通知が送信される。
- 全体OFF時に通知ルールを `registered` や `disabled` へ変えてしまい、ONへ戻しても送信されない。
- 詳細ペインに個別ON/OFF UIが残り、設定画面の全体設定と責務が混ざる。
- 既存DBに `notifications_enabled` カラムがなく起動に失敗する。

### スケール

全体OFF時は通知ジョブ一覧を取得しないため、大量の期限到来通知があってもOS通知送信ループへ入らない。

## トレードオフ

- 全体ON/OFFは分かりやすく誤操作が少ない一方、特定タスクだけ通知を止めることはできない。
- 通知ルールを変更せずdispatch入口で止めるため、ONへ戻した時の挙動は単純になる一方、設定状態とルール状態を別々に理解する必要がある。

## 代替案

タスク/サブタスク詳細で開始日通知・終了日通知を個別にON/OFFする。

不採用理由:

- 現時点の運用では詳細画面に個別設定を増やすより、設定画面で全体制御する方が分かりやすい。

## 受け入れ条件

- 設定画面で通知全体をON/OFFでき、状態が保存される。
- OFF中は期限到来通知がOSへ送信されない。
- OFF中も通知ルールは保持される。
- ONへ戻した後、条件を満たす通知ルールはdispatch対象に戻る。
- 右詳細ペインに通知個別ON/OFF UIがない。
- メモ本文や通知本文をログへ出さない。

## レビュー判断

フォローアップ付き承認。

- 通知時刻の任意設定と失敗履歴の一覧は別Issueで扱う。
- OSごとの実通知挙動は既存のWindows優先運用に合わせ、実機確認で継続確認する。
