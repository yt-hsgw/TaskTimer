# 028: 大量データで一覧とカレンダー表示を検証する

GitHub Issue: #72

Status: 完了。201件目以降へ到達できない問題は #131 で継続する。

## 目的

実務運用でタスク数が増えた場合でも、タスク一覧、今日、お気に入り、カレンダー、右詳細ペインが操作不能にならないことを確認する。

## スコープ

- 大量データ検証用のデータ作成手順を定義する。
- 生成したDBで主要Read Model相当のクエリ時間を測れるようにする。
- タスク一覧、今日、お気に入り、週/日/月カレンダー、右詳細ペインの表示時間を確認する。
- Windows runner上のヘッドレスChromeでPresentation描画時間を継続計測する。
- Presentationが全期間カレンダーや全タイマー履歴を読み込まないことを確認する。
- アクティブタイマーとアクティブポモドーロの復元が大量履歴に引きずられないことを確認する。
- ボトルネックが見つかった場合は、仮想リスト、ページング、Read Model改善のIssueへ分割する。

## スコープ外

- クラウド同期。
- 外部計測サービス。
- 実行時の分析送信。

## 設計レビュー

### データモデル

既存の `tasks`、`subtasks`、`timer_sessions`、`notification_rules` を使う。性能確認のために本番データモデルへテスト専用フィールドを追加しない。

### トランザクション境界

検証データ投入は開発・検証用スクリプトまたは手順で扱い、アプリ通常Use Caseのトランザクション境界を変更しない。通常操作では既存どおりApplication Use Caseが更新単位になる。

Read Model計測は `measure_large_dataset` binで行い、SQLite DBを読み取り専用の検証対象として扱う。計測結果はGitHub #72またはPRへテキストで記録し、DBファイル自体は添付しない。

### セキュリティ

検証データはサンプル文字列で作成し、実タスク名、メモ本文、SQLite DB、個人情報をIssueやPRへ添付しない。外部通信、分析SDK、新しいTauri権限は追加しない。

## トレードオフ

- 先に計測することでボトルネックを特定しやすい一方、実装前に完全な性能目標を固定するとUI改善の自由度が下がる。
- 仮想リストは大量データに強いが、完了セクションやサブタスク展開との相互作用が複雑になる。
- Read Modelを太らせるとUIの走査は減るが、Repository実装とクエリが複雑になる。

## 代替案

実データが増えてから性能改善する。

不採用理由:

- ローカルデスクトップアプリでは、利用者のDBが大きくなってから表示不能になると回復手段が少ない。
- 事前に性能上限と破綻シナリオを把握しておく方が、実務運用の信頼性を上げられる。

## 受け入れ条件

- 検証データ規模、作成手順、計測観点がdocsに記録されている。
- Windows環境で主要ビューが操作不能にならないことを確認している。
- ボトルネックが見つかった場合は、再現手順と改善Issueが作成されている。
- 実行時外部通信と追加Tauri権限を追加していない。

## 検証手順

詳細手順は [大量データ性能検証手順](../performance-large-dataset.md) を正とする。

標準検証データ:

- タスク: 5,000件。
- サブタスク: 20,000件。
- 停止済みタイマー履歴: 50,000件。
- アクティブポモドーロ: 1件。
- タスクリスト: 12件。

生成コマンド:

```bash
npm run perf:seed -- --force
```

Read Model計測:

```bash
npm run perf:measure
```

Windows runner計測:

```text
GitHub Actions > 大量データ性能検証 > Run workflow
profile=standard
fail_on_warning=true
```

Presentation描画計測:

```bash
npm run perf:ui -- --profile standard --fail-on-warning
```

標準UIプロファイルは、一覧集計5,000件に対して現行Tauri commandの取得上限と同じ200タスク、各4サブタスク、12リストを合成し、初期表示、今日、お気に入り、かんばん、週/日/月カレンダー、右詳細を操作する。SQLite 5,000件計測とPresentation 1ページ200件描画を組み合わせ、DBとReactのボトルネックを分離する。

Windows実機では、既存 `tasktimer.sqlite3` をバックアップしてから、生成した `tmp/perf/tasktimer-large.sqlite3` をアプリデータ配下へ配置する。

## 期待値

| 対象 | 期待値 |
| --- | --- |
| 起動 | ウィンドウ表示後5秒以内に操作可能。 |
| タスク一覧、今日、お気に入り | 切替後1秒以内に操作可能。 |
| 週/日カレンダー | 表示切替と前後移動が1.5秒以内に操作可能。 |
| 月カレンダー | 表示切替と前後移動が2秒以内に操作可能。 |
| 右詳細ペイン | タスク/サブタスク選択後1秒以内に表示。 |

## 現在の状態

- 2026-07-18に受け入れ条件を満たし、完了と判断した。
- 検証DB生成手段と計測手順は整備済み。
- Read Model相当のDB計測コマンドを追加済み。
- `active_pomodoro_lookup` をRead Model計測対象へ追加済み。
- GitHub-hosted Windows runnerでDB生成とRead Model計測を実行するworkflowを追加済み。
- `大量データ性能検証` workflowをmain上で標準プロファイル実行済み。run `29511423778` は5,000タスク / 20,000サブタスク / 50,000タイマー履歴でWARN 0、最大67ms。
- PR #132 のWindows runnerでPresentation smoke計測を実行済み。50タスク / 200サブタスク / 4リストでWARN 0、最大3,078ms。
- mainのrun `29647070007` でWindows標準プロファイルを実行済み。Read Modelは5,000タスク / 20,000サブタスク / 50,001タイマーセッションでWARN 0、最大78ms。Presentationは一覧集計5,000件 / 描画200タスク / 800サブタスクでWARN 0、最大1,893ms。
- `list_tasks` と `list_task_rows` は200件で打ち切られ、追加読み込み境界がないため、201件目以降へ到達できない。データ到達性の修正はGitHub #131へ分割した。
- ボトルネックが見つかった場合は、仮想リスト、ページング、カレンダー範囲クエリ改善、タイマー履歴表示上限のいずれかへ分割する。

## 危険ケース

- 今日ビューやお気に入りビューが全タスク/全サブタスクを毎回走査して描画停止する。
- 月カレンダーが表示範囲外のタスクまで読み込む。
- 右詳細ペインがタイマー履歴を無制限に表示しようとする。
- 大量通知ルールがある状態で再フォーカス時dispatchがUI操作を阻害する。

## レビュー判断

完了承認。

- 検証手順とWindows標準計測結果を記録し、改善実装はボトルネックごとに分割した。
- DBクエリ時間、Presentation描画時間、Tauri/WebView2実機時間を別の計測境界として扱う。
- 200件上限を仮想リストだけで隠さず、Repository/Applicationページングを #131 で設計する。
- Windows runnerでDB標準データと実React標準1ページの両方が閾値内であるため、Issue #72の性能検証を完了とする。Tauri/WebView2結合時の端末差はリリース前の実機確認で扱う。
