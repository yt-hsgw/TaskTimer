# 034: バックアップ/復元/エクスポートUIを追加する

GitHub Issue: #89

## 目的

設定画面から、ローカルバックアップ、復元、JSON/CSVエクスポートを実行できるようにする。

## スコープ

- 設定画面にデータ管理セクションを追加する。
- SQLiteバックアップ作成、SQLite復元、JSON/CSVエクスポートの操作導線を追加する。
- 復元前の確認メッセージを追加する。
- 個人データ含有の警告を表示する。
- 成功、失敗、キャンセル状態を表示する。

## スコープ外

- 自動バックアップ。
- クラウド保存先選択。
- 同期。

## 設計レビュー

### UI/UX

データ管理は設定画面配下に置く。通常のタスク作業ビューには出さず、危険操作として確認と注意喚起を明確にする。

保存先はファイル名指定ではなくフォルダ選択にする。SQLiteバックアップ、JSONエクスポート、CSVエクスポートの命名は既存Use Caseへ委譲する。

### トランザクション境界

UIはUse Caseを呼び出すだけにし、ファイル検証やDB入れ替え判断をPresentationへ持たせない。

復元成功後だけUIスナップショットを再取得する。バックアップ作成とエクスポートはタスクデータを変更しないため、一覧の再取得は行わない。

### セキュリティ

エラー文言はユーザー内容を含めない。ファイルパスも必要最小限にする。バックアップ/エクスポートファイルを公開場所へ添付しない注意を表示する。

Tauri権限はフォルダ選択に必要な `dialog:allow-open` に限定する。外部通信、クラウド保存先、自動アップロードは追加しない。

### 実装メモ

- 設定画面に「データ管理」セクションを追加する。
- データ管理操作中はボタンを無効化し、多重実行を防ぐ。
- 復元は復元元フォルダ選択後に確認メッセージを表示する。
- 成功、失敗、キャンセルを設定画面内の状態表示として返す。
- 成功表示は保存先/復元元の末尾名だけを表示し、フルパスは表示しない。

### 代替案

OSの保存ダイアログでファイル名まで指定させる案もある。

不採用理由:

- SQLiteバックアップはmanifest付きフォルダ形式であり、ファイル単体保存に見せると復元単位を誤解しやすい。
- 形式ごとの命名規則をUIへ重複実装すると、Application Use Caseとの責務境界が曖昧になる。

### トレードオフ

- フォルダ選択に限定することで、利用者が任意ファイル名を付ける柔軟性は下がる。
- 代わりに、バックアップ/エクスポート形式の一貫性と復元単位の分かりやすさを優先できる。

## 受け入れ条件

- 設定画面の導線が既存UIトーンに合っている。
- バックアップ/復元/エクスポートの成功、失敗、キャンセル状態が表示される。
- 復元時の確認メッセージと破損/不一致時のエラー文言がdocsと一致する。
- キーボード操作と狭い幅で操作不能にならない。

## 危険ケース

- 復元操作が軽い操作に見えて、利用者が現在DBを失う。
- エラー表示にタスク名やファイル内容が出る。
- 操作中に多重実行され、DB入れ替えが競合する。
