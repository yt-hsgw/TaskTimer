# 012: カレンダーと設定を左ナビ配下のビューへ移管する

GitHub Issue: #28

## 目的

現状常時表示している週カレンダーとローカル通知設定を、左ペインから開くビューとして整理する。

## スコープ

- 週カレンダーを中央ペインのカレンダービューへ移管する。
- カレンダー項目選択で右詳細ペインを開く。
- ローカル通知設定を設定ビューへ移管する。
- 通知再試行と通知状態を設定ビューで表示する。

## スコープ外

- 月表示、日表示。
- カレンダー上のドラッグ移動。
- OS将来時刻スケジューリング。

## 設計レビュー

### データモデル

週カレンダーは既存の日付範囲Read Modelを維持する。設定ビューは通知表示設定とUI設定を扱う。
カレンダー項目から詳細を開く選択状態はPresentationの一時状態であり、DBには保存しない。
サブタスクのカレンダー項目を選択した場合は、親タスク詳細を開き、タスク詳細内のサブタスク編集領域で確認できるようにする。

### トランザクション境界

- カレンダー表示は読み取り専用。
- 通知表示モード更新は既存 `UpdateNotificationPreference` を利用する。
- 通知再試行は既存の期限到来通知dispatchを明示操作として実行する。
- カレンダー項目選択はUI状態変更のみで、Application Use Caseを呼ばない。

### セキュリティ

設定画面に外部通信を有効化する設定を追加しない。
通知状態にはタスク名やメモ本文を表示せず、集計件数とエラーだけを表示する。

### 破綻シナリオ

- カレンダー項目選択時にDB更新を行うと、読み取り専用ビューの責務が崩れる。
- サブタスク項目の親タスク解決に失敗した場合、古い詳細ペインを表示し続けると利用者が別タスクを編集する危険がある。
- 設定ビューへ外部通信やクラウド同期の設定を追加すると、オフライン前提と矛盾する。
- 通知再試行を画面表示だけで暗黙実行すると、通知が繰り返し送られる可能性がある。

### スケール

週カレンダーは既存の週範囲Read Modelを使い、全タスクをカレンダー表示のために走査しない。
サブタスク項目の親解決は、詳細表示用に既に保持しているタスクツリー内で行う。

## トレードオフ

- カレンダーを常時表示しないことで一覧へ集中できるが、予定確認には1操作増える。
- 設定をビュー化すると画面は整理されるが、通知状態が常時見えなくなる。
- サブタスク項目で親タスク詳細を開くと実装は単純になるが、サブタスク単体の詳細フォーカスは後続改善になる。

## 代替案

週カレンダーを右ペインに置く。

不採用理由:

- 右ペインは詳細編集の場所として使うため、役割が混ざる。

## 受け入れ条件

- 左ペインからカレンダービューへ移動できる。
- 週カレンダーの前週/翌週移動が維持される。
- カレンダー項目選択で詳細ペインが開く。
- 設定ビューで通知表示タイプを変更できる。

## レビュー判断

フォローアップ付き承認。

- #28ではカレンダーと設定を左ナビ配下の中央ビューへ移管する。
- カレンダー項目はクリックで右詳細ペインを開く。
- サブタスク単体の詳細フォーカスは、右ペイン内のサブタスク編集が肥大化した時点で別Issue化する。
