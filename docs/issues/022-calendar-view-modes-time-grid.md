# 022: カレンダー週/日/月ビューと時間軸表示を実装する

GitHub Issue: #60

## 目的

カレンダー画面を実務で予定確認しやすいビューに拡張する。
本Issueでは日/月ビューの有効化と、週/日表示の時間軸化を扱う。

## スコープ

- `週`、`日`、`月` の表示切替を実際に有効化する。
- 週表示を月曜から日曜の横並びにし、一般的な始業時間からの時間軸を表示する。
- 開始予定、期限、実行中タイマーを時間軸上で判別できるようにする。
- サブタスク項目は親タスクとの関係が分かる表示にする。
- カレンダー項目選択で右詳細ペインを開く既存導線を維持する。

## スコープ外

- カレンダー上のドラッグ移動。
- 外部カレンダー同期。
- OSグローバル通知や自動更新通信。

## 設計レビュー

### データモデル

既存のタスク/サブタスク日付とアクティブタイマーを使う。
時間軸表示では日付のみの予定を時間指定予定と誤認させないため、開始予定/期限は「日付のみ」行へ配置する。
実行中タイマーは `started_at` から表示用時刻を派生し、9:00から18:00の時間軸へ配置する。
サブタスクは親タスク名をRead Modelに含め、カレンダー項目内で関係を表示する。

### トランザクション境界

カレンダー表示は読み取り専用。
表示切替はPresentation状態であり、DB更新を行わない。
将来UI設定として最後の表示モードを保存する場合は、業務データとは分けたUI設定Use Caseで扱う。
Repositoryは開始日・終了日の範囲取得を提供し、コマンド入力は日付形式、開始/終了順、93日以内の範囲を検証する。

### セキュリティ

外部カレンダー連携や通信は追加しない。
カレンダー項目にもメモ本文は表示しない。
タイトル表示は既存の通知プライバシー設定とは独立したアプリ内表示として扱う。

### 破綻シナリオ

- 日付だけのタスクを時間軸へ無理に配置して、利用者が具体的な時刻予定と誤認する。
- サブタスクが親タスクと無関係に見え、どの作業の一部か判断できない。
- 月表示で大量タスクを全件描画し、画面が重くなる。
- 表示モード切替時に選択中詳細が古いタスクのまま残る。

### スケール

週/日ビューは表示期間でDB範囲を絞る。
月ビューは月表示グリッドに必要な最大6週程度の範囲に絞り、1日あたり3件まで表示し、残件数を表示する。
全タスクや全タイマー履歴をカレンダー描画のために読み込まない。

## トレードオフ

- 日付のみのMVPデータで時間軸UIを作ると、時刻精度は高くないが業務上の見通しは改善する。
- 日/月ビューを先に有効化すると操作の期待値が上がるため、空状態や未実装状態を残さず一通りの表示を実装する必要がある。
- Read Modelをビューごとに分けると最適化しやすいが、Use Caseが増える。

## 代替案

週表示だけを維持し、日/月切替を追加しない。

不採用理由:

- UI上の将来導線がなく、カレンダー画面の完成像が分かりにくい。
- 実務利用では日単位、月単位の俯瞰が必要になる。

## 受け入れ条件

- カレンダーで週/日/月を切り替えられる。
- 週表示で月曜から日曜が横並びになり、始業時間からの時間軸が分かる。
- 日付のみのタスクが、時刻指定予定と誤認されにくい表示になる。
- サブタスク項目から親タスクとの関係が分かる。
- 768px未満でも操作不能な重なりが発生しない。

## レビュー判断

承認。

- 週/日/月ビューを実表示として有効化する。
- 日付のみ予定は「日付のみ」行へ置き、時間指定と誤認しない表示にする。
- カレンダーRead Modelは範囲取得へ拡張し、サブタスクの親名と実行中タイマーの表示用時刻を含める。
- ドラッグ移動、外部カレンダー同期、通知スケジューリング方式変更は引き続きスコープ外。
