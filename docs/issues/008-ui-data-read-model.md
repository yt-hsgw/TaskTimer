# 008: UI/UX再設計用データモデルとRead Modelを整備する

GitHub Issue: #27

## 目的

左ナビ、タスク一覧、右詳細ペインに必要な情報を、Presentationへドメインルールを漏らさず提供する。

## スコープ

- `task_lists` の設計。
- `tasks.is_favorite` の設計。
- タイマー目標時間の設計。
- UI設定保存の設計。
- タスク一覧Read Modelの設計。
- サブタスク進捗集計の設計。

## スコープ外

- 画面実装。
- タイマー一時停止/再開の実装。
- 繰り返し設定の実装。

## 設計レビュー

### データモデル

タスク一覧UIで必要な情報を正規化データから集計する。進捗表示のためにUIが全サブタスクを毎回走査しないよう、Repositoryで集計済みDTOを返す。

### トランザクション境界

- リスト作成は1トランザクション。
- お気に入り切り替えはタスク1件更新。
- UI設定は業務データと分けて更新する。

### セキュリティ

- UI設定にタスク名やメモ本文を保存しない。
- Read Model取得時にメモ本文を不要に返さない。

## トレードオフ

- 専用Read ModelはDTOが増えるが、UIが巨大データを走査するより安全。
- `task_lists` を追加するとマイグレーションが必要だが、左ペインのリスト一覧を明確に表現できる。

## 代替案

既存の `list_tasks` を拡張し、UI側で進捗やお気に入り表示を組み立てる。

不採用理由:

- 大量タスク時にUI処理が重くなりやすい。
- 表示ルールがPresentationへ寄る。

## 受け入れ条件

- タスク一覧に必要な行DTOが定義されている。
- サブタスク進捗がDBまたはRepositoryで集計される。
- お気に入り状態がタスク単位で保存できる。
- 既存タスクが初期リストに所属する移行方針がある。
