# 008: UI/UX再設計用データモデルとRead Modelを整備する

GitHub Issue: #27

## 目的

左ナビ、タスク一覧、右詳細ペインに必要な情報を、Presentationへドメインルールを漏らさず提供する。

## スコープ

- `task_lists` の設計。
- `tasks.is_favorite` の設計。
- タイマー目標時間の設計。
- UI設定保存の設計。
- タスク一覧Read Modelの設計。
- サブタスク進捗集計の設計。

## スコープ外

- 画面実装。
- タイマー一時停止/再開の実装。
- 繰り返し設定の実装。

## 設計レビュー

### データモデル

タスク一覧UIで必要な情報を正規化データから集計する。進捗表示のためにUIが全サブタスクを毎回走査しないよう、Repositoryで集計済みDTOを返す。

追加する永続データ:

- `task_lists`: 左ペインのリスト一覧。初期リストは `id = default`, `name = タスク`。
- `tasks.list_id`: タスク所属リスト。既存タスクは初期リストへ移行する。
- `tasks.is_favorite`: お気に入り状態。MVPではタスク単位だけを対象にする。
- `tasks.timer_target_seconds`, `subtasks.timer_target_seconds`: 目標タイマー時間。未設定は `NULL`。
- `ui_preferences`: 左ペイン開閉、最後のビュー、最後のリストなどのローカルUI状態。

タスク一覧Read Model:

- `TaskRowDto` はタイトル、状態、お気に入り、期限、目標タイマー時間、サブタスク件数、完了済みサブタスク件数、アクティブタイマー対象を返す。
- 一覧行ではメモ本文を返さない。メモは右詳細ペイン用の取得で扱う。
- 未完了タスクを先、完了タスクを後に並べる。Presentationはこの結果を未完了セクションと完了セクションへ分けられる。
- サブタスク進捗はRepositoryで集計し、Presentationは全サブタスク走査に依存しない。

初期リスト移行:

- 新規DBでは初期スキーマに `task_lists` と関連カラムを含める。
- 既存DBでは起動時migrationで不足カラムを検出して追加し、`tasks.list_id IS NULL` の既存タスクを `default` にバックフィルする。
- SQLiteの `ALTER TABLE` 制約により既存DBへ後付けした `tasks.list_id` には外部キー制約を追加しない。アプリケーション層で初期リスト存在を保証し、新規DBでは外部キー制約を持つ。

### トランザクション境界

- リスト作成は1トランザクション。
- お気に入り切り替えはタスク1件更新。
- UI設定は業務データと分けて更新する。
- タスク作成時は初期リスト存在確認とタスクINSERTを同一トランザクション内で扱う。
- Read Model取得は読み取り専用で副作用を持たない。

### セキュリティ

- UI設定にタスク名やメモ本文を保存しない。
- Read Model取得時にメモ本文を不要に返さない。
- 追加commandはSQLプレースホルダを使い、リストIDを文字列結合しない。
- アプリ実行時の外部通信、追加権限、秘密情報を追加しない。

## トレードオフ

- 専用Read ModelはDTOが増えるが、UIが巨大データを走査するより安全。
- `task_lists` を追加するとマイグレーションが必要だが、左ペインのリスト一覧を明確に表現できる。

## 代替案

既存の `list_tasks` を拡張し、UI側で進捗やお気に入り表示を組み立てる。

不採用理由:

- 大量タスク時にUI処理が重くなりやすい。
- 表示ルールがPresentationへ寄る。

## 受け入れ条件

- タスク一覧に必要な行DTOが定義されている。
- サブタスク進捗がDBまたはRepositoryで集計される。
- お気に入り状態がタスク単位で保存できる。
- 既存タスクが初期リストに所属する移行方針がある。

## レビュー結果

- 指摘事項: UI実装前の土台としてRead Model commandを追加する。お気に入り切り替え、リスト作成、UI設定更新は別Issueで実装する。
- 破綻シナリオ: migration途中失敗時は起動時にエラーを返す。部分適用後も再実行できるよう、不足カラム検出と `INSERT OR IGNORE` を使う。
- スケール懸念: タスク一覧は上限件数をRepositoryで制限し、サブタスク進捗はDB集計に寄せる。
- セキュリティ懸念: 一覧Read Modelにメモ本文を含めず、UI設定にもユーザー内容を保存しない。
- テスト不足: 画面での表示確認はUI改修Issueで実施する。本IssueではmigrationとRepository集計を単体テストで確認する。
- 判断: フォローアップ付き承認。
