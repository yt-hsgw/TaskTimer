# 029: ローカルデータのバックアップとエクスポート方針を設計する

GitHub Issue: #73

## 目的

TaskTimerのローカルSQLiteデータを、端末故障、移行、誤削除に備えて安全に退避できる方針を設計する。

## スコープ

- バックアップ対象を定義する。
- エクスポート形式を検討する。
- 復元時のマイグレーション、バージョン不一致、破損DBの扱いを定義する。
- UI実装を行うか、まずは運用手順として提供するかを決める。
- 個人データをIssue、PR、ログへ出さない運用を明記する。

## スコープ外

- クラウド同期。
- 外部ストレージ連携。
- 自動バックアップのための外部通信。
- チーム共有や共同編集。

## 設計レビュー

### データモデル

バックアップ対象の第一候補はローカルSQLite DBとローカル設定。タスク名、サブタスク名、メモ本文、通知ルール、タイマー履歴は個人データとして扱う。

### トランザクション境界

アプリ内にバックアップ機能を実装する場合は、書き込み中のDBコピーを避ける境界が必要になる。採用案では、バックアップ開始前に書き込みUse Caseと競合しない設計を明示する。

### セキュリティ

エクスポートファイルには個人データが含まれる可能性がある。Issue、PR、Release notes、Discussionsへ添付しない。暗号化やパスワード保護を行う場合でも、秘密情報をリポジトリへ保存しない。

## トレードオフ

- SQLiteファイルコピーは復元しやすい一方、アプリバージョンやマイグレーションとの整合確認が必要になる。
- JSON/CSVエクスポートは人間が確認しやすい一方、完全復元や通知ルール、タイマー履歴の再現が難しい。
- UIから提供すると利用者に分かりやすいが、ファイル選択、権限、破損時エラー表示の設計が必要になる。
- 手順書から始めると実装量は小さいが、非技術利用者には扱いづらい。

## 代替案

バックアップ機能を追加せず、OSやユーザー自身のファイルバックアップに任せる。

不採用理由:

- TaskTimerはローカル保存が正であり、端末故障や移行時のデータ保護方針を説明できないと実務利用の不安が残る。
- 少なくとも「何を退避すればよいか」をdocsに明記する必要がある。

## 受け入れ条件

- 採用案、代替案、トレードオフがdocsへ記録されている。
- バックアップ/復元で扱うファイルと扱わないファイルが明確。
- 破損時やバージョン不一致時のエラー表示方針が定義されている。
- 実装が必要な場合は、Use Case、Repository、UIのIssueへ分割されている。

## 危険ケース

- ユーザーがSQLiteファイルをIssueへ添付して個人データを公開してしまう。
- DBコピー中にアプリが書き込み、バックアップが破損する。
- 古いDBを新しいアプリへ復元してマイグレーションに失敗する。
- エクスポートにメモ本文が含まれることを利用者が理解していない。

## レビュー判断

フォローアップ付き承認。

- まずは方針と運用手順を設計し、UI実装は採用案が固まってから分割する。
