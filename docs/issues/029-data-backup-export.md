# 029: ローカルデータのバックアップとエクスポート方針を設計する

GitHub Issue: #73

## 目的

TaskTimerのローカルSQLiteデータを、端末故障、移行、誤削除に備えて安全に退避できる方針を設計する。

## スコープ

- バックアップ対象を定義する。
- エクスポート形式を検討する。
- 復元時のマイグレーション、バージョン不一致、破損DBの扱いを定義する。
- UI実装を行うか、まずは運用手順として提供するかを決める。
- 個人データをIssue、PR、ログへ出さない運用を明記する。

## スコープ外

- クラウド同期。
- 外部ストレージ連携。
- 自動バックアップのための外部通信。
- チーム共有や共同編集。

## 設計レビュー

### データモデル

バックアップ対象の正は `tasktimer.sqlite3` とする。現時点の通知設定とUI設定は `notification_preferences` / `ui_preferences` に保存されているため、別設定ファイルは対象にしない。

タスク名、サブタスク名、メモ本文、通知ルール、通知失敗履歴、タイマー履歴は個人データとして扱う。

### トランザクション境界

採用案では、完全復元用の正をSQLiteバックアップにする。アプリ内にバックアップ機能を実装する場合は、単純なファイルコピーではなくSQLiteの一貫したスナップショットを作る。復元は一時DBで検証とマイグレーション可否確認を行い、成功後に現DBを退避して入れ替える。

### セキュリティ

エクスポートファイルには個人データが含まれる可能性がある。Issue、PR、Release notes、Discussionsへ添付しない。暗号化やパスワード保護を行う場合でも、秘密情報をリポジトリへ保存しない。

## 採用案

詳細は [ローカルデータのバックアップとエクスポート方針](../data-backup-export.md) と [ADR 0006](../adr/0006-local-backup-export-policy.md) を正とする。

- フル復元用の正はSQLiteバックアップにする。
- SQLiteバックアップは `tasktimer.sqlite3` と `backup-manifest.json` の組み合わせにする。
- JSON/CSVは閲覧、監査、他ツール移行の補助形式として扱い、完全復元には使わない。
- 復元時は `PRAGMA integrity_check`、必須テーブル、マイグレーション適用可否を確認する。
- 現在のアプリが理解できない新しいDBは復元を拒否する。
- UIは設定画面の「データ管理」セクションとして後続実装する。

## バックアップ対象

対象:

- `tasktimer.sqlite3`
- `backup-manifest.json`
- manifestに記録するapp version、作成日時、プラットフォーム、整合性確認結果

対象外:

- GitHub Issue、PR、Release artifactへのDB添付
- OS通知許可状態
- アプリのインストーラー/実行ファイル
- ログファイル
- クラウド保存先設定

## 破損・バージョン不一致時の表示方針

| 状態 | 方針 |
| --- | --- |
| SQLiteとして開けない | 復元を拒否する。 |
| `PRAGMA integrity_check` が `ok` でない | 復元を拒否する。 |
| 必須テーブルがない | 復元を拒否する。 |
| 古いDBだが既存マイグレーションで更新可能 | 一時DB上でマイグレーション後に復元する。 |
| 現在のアプリが理解できない新しいDB | 復元を拒否し、新しいTaskTimerで作成された可能性を説明する。 |
| 復元処理中に失敗 | 既存DBを保持し、ユーザー内容をログへ出さない。 |

## トレードオフ

- SQLiteファイルコピーは復元しやすい一方、アプリバージョンやマイグレーションとの整合確認が必要になる。
- JSON/CSVエクスポートは人間が確認しやすい一方、完全復元や通知ルール、タイマー履歴の再現が難しい。
- UIから提供すると利用者に分かりやすいが、ファイル選択、権限、破損時エラー表示の設計が必要になる。
- 手順書から始めると実装量は小さいが、非技術利用者には扱いづらい。

## 代替案

バックアップ機能を追加せず、OSやユーザー自身のファイルバックアップに任せる。

不採用理由:

- TaskTimerはローカル保存が正であり、端末故障や移行時のデータ保護方針を説明できないと実務利用の不安が残る。
- 少なくとも「何を退避すればよいか」をdocsに明記する必要がある。

## 受け入れ条件

- 採用案、代替案、トレードオフがdocsへ記録されている。
- バックアップ/復元で扱うファイルと扱わないファイルが明確。
- 破損時やバージョン不一致時のエラー表示方針が定義されている。
- 実装が必要な場合は、Use Case、Repository、UIのIssueへ分割されている。

- ユーザーがSQLiteファイルをIssueへ添付して個人データを公開してしまう。
- DBコピー中にアプリが書き込み、バックアップが破損する。
- 古いDBを新しいアプリへ復元してマイグレーションに失敗する。
- エクスポートにメモ本文が含まれることを利用者が理解していない。
