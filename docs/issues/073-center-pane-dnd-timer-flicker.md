# 073 中央ペインのD&D移動とタイマー操作ちらつきを修正する

GitHub Issue: #187

## 背景

中央ペインでタスクや予定ブロックをD&Dした時に、移動直後の表示が不安定になるケースがある。また、タスクの再生ボタンを押すと、タスクページ全体の再取得と詳細mutation扱いにより、周辺ボタンまでちらついて見える。

## 対応範囲

- かんばんのタスクD&Dが列移動として成立することを回帰確認する。
- カレンダーの月表示で、予定ブロック移動後に期間を維持し、複数日表示が連結されるようにする。
- カレンダーの移動/期間変更は保存前に楽観表示を即時反映する。
- タスクタイマー開始、一時停止、再開、終了ではタスクページ全体を再取得しない。
- タスクタイマー操作は詳細mutationから分離し、周辺ボタンをまとめてdisabledにしない。

## 非対象

- タスク行三点メニュー追加。
- タイムラインの蓋閉じ。
- ポモドーロ画面のUI改善。
- タスクタイマーのDBモデル変更。

## 設計方針

### D&D

カレンダーのD&DはPresentation上のPointer/DragEventから移動先を計算し、Application Use Caseには構造化した予定移動だけを渡す。保存前の予測表示はPresentation stateとして扱い、保存成功後にRead Modelを再同期する。

```mermaid
flowchart LR
  Drag["予定ブロックD&D"] --> Preview["移動後プレビュー"]
  Preview --> Pending["楽観表示"]
  Pending --> UseCase["MoveScheduledWorkItem"]
  UseCase --> Refresh["表示中範囲を再同期"]
```

### タイマー

通常タイマーの開始/一時停止/再開/終了は、Use Caseの戻り値で `activeTimer` を局所更新する。通知同期は維持するが、タスクページ全体の再取得はカウントダウン期限到来など、タスク状態が実際に変わるタイミングへ限定する。

## 設計理由

- D&D直後の「移動元が一瞬戻る」見え方は、保存完了を待ってから表示更新するほど発生しやすい。
- タイマー操作で全タスク行を再取得すると、ユーザーが押していないボタンまでdisabled/再描画され、ちらつきとして知覚される。
- タイマーの正はDBに残すが、操作直後の表示はUse Case戻り値で十分に同期できる。

## トレードオフ

- タイマー開始中の `TaskRow.isTimerActive` は局所パッチで補正するため、ページ外の行は次回読み込みまで更新されない。
- タイマー期限到来時はタスク状態や通知が関わるため、従来どおりTaskPage再同期を残す。
- カレンダーの楽観表示を即時flushすると、保存失敗時に元表示へ戻す責務がPresentationへ残る。

## 代替案

全操作後に `loadSnapshot` 相当でアプリ全体を再取得する。

不採用理由: 実装は単純だが、D&D後のちらつき、スクロール位置喪失、ボタンの一斉再描画が悪化する。

## 危険ケース

- D&D保存失敗後に楽観表示が残る。
- 月表示の週またぎ複数日予定で連結クラスが外れ、別々の予定に見える。
- タイマー停止後に一覧行の `実行中` 表示だけ残る。
- タイマー操作を局所更新しすぎて、期限到来時の完了通知や状態更新が反映されない。
- 連打で通常タイマーの単一アクティブ制約をPresentation側でも破って見せる。

## セキュリティと権限

- 新しい外部通信、ファイルアクセス、OS権限、Tauri capabilityは追加しない。
- タスク名、メモ、通知本文をログへ出さない。
- D&Dで保存する日付/時刻は既存Application Use Caseの検証境界を通す。

## 受け入れ条件

- かんばんD&Dでタスクを別列へ移動できる。
- カレンダー月表示で予定ブロックをD&Dし、期間を維持したまま日付変更できる。
- D&D後に詳細ペインが自動で開かない。
- D&D直後に移動元へ一瞬戻る表示がない。
- タイマー再生ボタン押下時にタスクページ全体を再取得しない。
- タイマー再生ボタン押下時に無関係なボタンがまとめてちらつかない。
- 大量データUI計測のsmoke/standardで警告が増えない。
