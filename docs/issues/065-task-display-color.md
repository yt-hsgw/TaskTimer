# 065 タスク表示色をリスト色から分離する

GitHub Issue: #159

## 背景

現在の色はリスト単位であり、タスク詳細からリスト自体の色を変更している。利用者は所属リストと個別タスク色を別に設定し、カレンダーで両方を識別したい。

## データモデル

- `tasks.color_token` をnullableで追加する。
- nullは「所属リスト色を継承する」を意味する。
- 値は `green`、`blue`、`amber`、`rose`、`violet`、`gray` に限定する。
- サブタスクは親タスクの実効色を継承する。
- リスト色は左ペインのリスト編集、タスク色は親タスク詳細で変更する。

## 表示規則

- 実効タスク色は `task.color_token ?? list.color_token` とする。
- カレンダー項目の左端アクセントはリスト色、本文は実効タスク色とする。
- タスク色未設定時は左端と本文が同じリスト色になる。
- 色だけに意味を持たせず、リスト名、種別、タイトルのテキストを維持する。

## トランザクション境界

タスク色変更は `UpdateTask` の同一トランザクションで保存する。Read Model、JSON/CSVエクスポート、旧DBマイグレーションを同じ変更で更新する。

## トレードオフ

- タスクごとの視認性は上がるが、色の意味が増える。
- nullable継承によりリスト色変更が未指定タスクへ即時反映される一方、個別色タスクには反映されない。

## 代替案

タスク色を必須にし、作成時にリスト色をコピーする。

不採用理由: 後からリスト色を変更しても既存タスクへ反映されず、継承意図が失われる。

## セキュリティと危険ケース

- 任意CSS、16進色、URLを受け入れず、許可済みトークンだけをDomain/Application/DBで検証する。
- 不正な旧値は読込時に既定色へ黙って変換せず、移行またはエラーとして検出する。
- タスク移動時に個別色を消さない。
- エクスポートに継承状態と個別色を区別して出力する。

## 受け入れ条件

- リスト色とタスク色を別の場所で変更できる。
- タスク色未指定時にリスト色を継承する。
- カレンダーで左端のリスト色と本文のタスク色を確認できる。
- 旧DB移行、更新、所属変更、JSON/CSV、許可値違反のテストが成功する。
