完了条件:

## Phase 1: プロジェクトスキャフォールド

成果物:

- Tauriプロジェクト構成。
- React + TypeScriptフロントエンド。
- Rust command境界。
- SQLiteマイグレーション実行器。
- 依存関係レビュー。

完了条件:

- macOS/Windowsの開発環境でアプリが起動する。
- 設定上、アプリ実行時の外部通信が無効化されている。
- 空データベースを初期化できる。
- 未決事項がプロダクト判断として反映されている。

現在のスキャフォールド状態:

- Tauri、Vite、React、TypeScriptのファイルは配置済み。
- Rust command境界はヘルスチェック、週カレンダー取得、アクティブタイマー取得、通知表示設定取得、タスク/サブタスク作成、タイマー開始/停止まで配置済み。
- SQLite初期マイグレーションとRepository実装は配置済み。
- `CreateTask`、`CreateSubtask`、`StartTimer`、`StopActiveTimer` はApplication Use Case経由で実行し、入力検証とトランザクション境界を持つ。
- タスク/サブタスク一覧は上限付きの読み取りRepositoryから取得し、UIはモックデータではなくSQLite由来の状態を表示する。
- 実行時HTTP、自動更新、分析、リモートアセット機能は設定していない。

## Phase 2: ドメインと永続化

成果物:

- Taskドメインモデル。
- Subtaskドメインモデル。
- TimerSessionドメインモデル。
- SQLite Repository。
- Application Use Case。

完了条件:

- ドメインユニットテストが通る。
- SQLite Repositoryテストが通る。
- 単一アクティブタイマー制約がApplicationとDBの両方で守られる。

設計判断:

- Use Caseは未検証入力を検証済みコマンドへ変換する境界として実装する。
- SQLite実装は1 Use Caseにつき1トランザクションを開始し、親存在確認、状態確認、タイマー作成/停止、対象状態更新を同一トランザクションに閉じる。
- OS通知などの副作用は今回のUse Caseには含めず、DBコミット後副作用として後続Issueで扱う。

トレードオフ:

- Tauri commandから直接SQLを書かず保守性を優先する一方、Application層の型とDTOが増える。
- `tasks` と `subtasks` を分けているため作成処理は重複するが、親タスクとサブタスクの意味を明確に保てる。

## Phase 3: コアUI

成果物:

- タスクリスト。
- タスク詳細。
- サブタスク編集。
- アクティブタイマー操作。
- 週カレンダー画面。

完了条件:

- ユーザーがMVP受け入れ条件を手動で完走できる。
- メモ本文をHTMLとして描画しない。

現在の到達点:

- タスク作成、サブタスク作成、タイマー開始/停止ボタンはTauri command経由でApplication Use Caseへ接続済み。
- 週カレンダーはSQLiteの週範囲クエリ結果を表示する。
- タスク一覧とサブタスク一覧はSQLiteの読み取りRepository結果を表示し、アプリ再起動後も保存済みデータを読める。
- 親タスク完了時の確認、サブタスク完了、タスク/サブタスク削除はApplication Use Caseへ接続済み。
- 編集とアーカイブは後続Issueで扱う。

設計判断:

- PresentationはSQLiteやトランザクションを直接扱わず、Tauri Gateway経由でApplication commandを呼ぶ。
- タスク一覧はMVPの安全上限として200件に制限し、週カレンダーは日付範囲で絞り込む。
- メモ本文はReactの通常テキスト描画と `white-space: pre-wrap` で表示し、HTMLとして描画しない。
- 未完了サブタスクがある親タスクの完了確認はUIで行い、Application Use Caseでも確認フラグを検証する。
- 削除時は関連タイマー履歴と通知ルールを同一トランザクションでソフト削除し、開始中タイマーを通常検索から除外する。

代替案:

UI専用にすべてのタスク、サブタスク、タイマー履歴を一括ロードする案もある。

利点:

- フロント側で自由に集計しやすい。

欠点:

- データ増加時に初期表示が重くなり、タイマー操作の応答性も落ちやすい。

決定: MVPではタスクツリーの上限付き読み取りと週カレンダーの範囲読み取りを分ける。

## Phase 4: 通知

成果物:

- 通知ルールUI。
- OS通知アダプター。
- 登録状態と再試行処理。

完了条件:

- macOS/Windowsでローカル通知が動く。
- 通知登録失敗がUIで分かり、再試行できる。

現在の到達点:

- 通知表示モードはSQLiteへ保存できる。
- タスク/サブタスク作成時に開始予定日・期限から通知ルールを作成する。
- 期限到来済み通知ルールはTauri notification plugin経由でOS通知へ送信する。
- 送信成功時は `registered`、失敗時は `failed` と `last_error` を保存する。
- 設定画面でdispatch結果と失敗メッセージを確認でき、再試行できる。

設計判断:

- `generic` モードでは対象タイトルをOS通知adapterへ渡さない。
- メモ本文は通知本文に含めない。
- MVPではアプリ起動中または再読み込み時に期限到来通知をdispatchし、OSへの将来時刻スケジューリングは後続改善とする。

## Phase 6: UI/UX再設計

成果物:

- UI/UX再設計仕様。
- 左ナビゲーションペイン。
- 中央タスク一覧ビュー。
- 右タスク詳細ペイン。
- カレンダービュー移管。
- 設定ビュー移管。
- タスク一覧Read Model。
- お気に入り、完了セクション、サブタスク進捗表示。
- レスポンシブ幅ごとのレイアウト。

完了条件:

- [UI/UX再設計仕様](ui-ux-redesign.md) の受け入れ条件を満たす。
- 既存のタスク/サブタスク/タイマー/通知Use Case境界をUIへ漏らさない。
- 左ペインの `Ctrl+B` とタスク追加の `Ctrl+N` が動作する。
- 週カレンダーと通知設定が新しいビューへ移管される。
- 768px未満でも、一覧、詳細、設定、カレンダーの主要操作が破綻しない。

設計判断:

- 実装はデータモデルとRead Modelから始める。
- Microsoft To Doの画面構造は参考にするが、色、タイマー体験、カレンダー体験はTaskTimer独自にする。
- 右ペインは基本非表示にし、タスク選択時に詳細を開く。
- タイマー一時停止/再開、繰り返し設定、リスト管理はデータモデル影響が大きいため、UIだけ先行実装しない。

推奨分割:

1. データモデルとRead Modelを拡張する。
2. 左ナビゲーションとApp Shellを作る。
3. 中央タスク一覧を新仕様へ置き換える。
4. 右詳細ペインへ詳細操作を移管する。
5. カレンダーと設定をビューとして移管する。
6. タイマー一時停止/再開、繰り返し、通知個別設定を追加する。

危険ケース:

- UIだけ先に作り、リスト、お気に入り、タイマー一時停止の正がPresentationに寄る。
- 完了済みタスク移動や進捗バーが、DB状態と不一致になる。
- 右ペインの削除操作が狭い画面で誤操作される。
- カレンダー移管時に既存の週範囲クエリが全件読み込みへ退化する。
